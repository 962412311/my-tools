# 01-workflow

## 目的
本文件定义通用执行流程，适用于开发、文档整理、规范建设、知识沉淀等工作。

---

## 标准流程

### 第一步：识别任务类型
收到任务后，先判断它属于哪一类：
- 开发任务
- 文档任务
- 知识管理任务
- Git / 部署任务
- 混合型任务

如为混合型任务，按主任务优先，并补充读取相关模块。

### 第二步：读取规范
开始正式执行前，至少完成：
1. 阅读 `~/.codex/GLOBAL-AGENT.md`
2. 阅读 `~/.codex/global-rules/core/03-priority.md`
3. 读取对应任务模块
4. 如当前工作目录存在 `.agent/local/90-project-specific.md`，优先读取

未完成这一步，不进入正式修改。

### 第三步：理解现状
在动手前，先快速回答：
- 目标是什么
- 当前状态是什么
- 约束是什么
- 成功标准是什么
- 相关文件、目录、模块、文档在哪里

### 第四步：提出最合理方案
遇到模糊需求时，不要立刻把问题全部丢回给用户。默认做法：
1. 基于上下文给出最合理的方案
2. 明确关键假设
3. 标出可调整项
4. 再给替代方案或可选分支

### 第五步：执行最小闭环
优先完成最小可用闭环，再考虑扩展。避免一开始就展开大面积重构或大范围结构调整。

### 第五步半：简单子任务委派
如果当前会话已经明确允许委派或使用子代理，则：
- 对明确、低风险、边界清晰、不会阻塞主线的非复杂子任务，默认优先委派给 `GPT-5.3-Codex-Spark`
- 对复杂设计、关键路径决策、高风险修改、强上下文耦合任务，不为了委派而委派
- 委派前先明确主线当前步骤和子任务边界，避免主线与子任务重复劳动

### 第六步：主动验证
修改后主动做验证。根据任务类型选择：
- test
- lint
- build
- typecheck
- 最小人工流程验证
- 链接检查
- 目录检查
- 文档可读性检查

### 第七步：报告结果
汇报结果时优先包含：
- 做了什么
- 为什么这样做
- 影响范围
- 验证情况
- 未完成项与风险

---

## 遇到模糊需求时的默认处理
除非存在真实风险，否则不要只回复问题。先给一版最合理方案，再说明：
- 我基于什么假设这样做
- 哪些地方可以调整
- 哪些地方需要用户后续补充

---

## 遇到规范缺失时的处理
不要在无规范状态下随意扩展结构。默认步骤：
1. 识别缺失点
2. 提出最小补充规则
3. 先补规则
4. 再落地实践

---

## 遇到冲突时的处理
按 `~/.codex/global-rules/core/03-priority.md` 执行，不自行创造“折中规则”。

---

## 禁止行为
- 未读规范就开始正式修改
- 边做边临时发明一套新结构
- 用表面修补替代根因修复
- 因为赶时间而跳过关键验证
- 把明显能自行推断的信息反复向用户确认
