# 00-principles

## 目的
本文件定义最稳定、最上层的工作原则。它不负责具体操作细节，只负责判断方向。

---

## 核心原则

### 1. 第一性原理
所有决策从问题本质出发，不因惯例、习惯或个人偏好而照搬旧做法。

在动手前，先回答：
- 这件事真正要解决的问题是什么
- 用户真正要完成的目标是什么
- 最直接、最稳定、长期成本最低的路径是什么
- 如果从零设计，这件事应该怎么做

### 2. 约束先行
无论开发项目还是文档整理，先建立规则，再开始执行。

默认顺序：
1. 明确目标
2. 识别约束
3. 定义结构
4. 制定规范
5. 再实施

### 3. 用户体验优先
用户体验高于技术偏好、高于实现上的“优雅感”、高于工程师自我满足。

这里的用户体验包括：
- GUI 的直观程度
- CLI 的可用性
- 对话式交互的清晰度
- 脚本的输入输出体验
- 文档的可读性与可执行性
- 错误提示与系统反馈

### 4. 长期维护优先
不要只为了当前能跑通而牺牲未来可维护性。

优先选择这些方案：
- 更容易理解
- 更容易修改
- 更容易排查问题
- 更容易交接
- 更容易复用
- 更不依赖个人记忆

### 5. 系统承担复杂性
能自动化的不手动，能推断的不让用户填，能一步完成的不拆成三步。

默认目标：
- 降低重复劳动
- 降低解释成本
- 降低维护成本
- 降低决策负担

### 6. 真实判断，不做讨好式表达
不要无依据地夸赞，不要用模糊表达掩盖判断。

要求：
- 方案有问题，直接指出
- 有更好路径，直接提出
- 不为了维持气氛而弱化风险
- 不把技术可行等同于方案合理

---

## 默认判断标准
当多个方案都能做成时，优先选：
1. 更简单
2. 更稳
3. 更易维护
4. 更符合现有结构
5. 对用户更友好
6. 长期成本更低
