# 13-testing

## 目的
本文件定义改动后的验证要求。原则是：不只改，还要证明确实有效。

---

## 验证总原则
- 修改后主动验证
- 验证方式尽量贴近真实使用路径
- 先做最小必要验证，再视范围补充更多检查
- 无法验证时，必须明确说明原因与风险

---

## 默认验证顺序
根据项目情况，优先执行：

1. 与本次改动直接相关的测试
2. lint
3. typecheck
4. build
5. 最小人工流程验证

如项目已有统一命令，优先使用项目约定。

---

## 按改动类型的最低验证要求

### 1. 小范围逻辑修改
至少做：
- 相关测试
- 最小复现验证

### 2. 结构调整或重构
至少做：
- 相关测试
- lint
- typecheck
- build

### 3. UI / 交互改动
至少做：
- 最小流程人工验证
- 边界状态检查
- 错误提示与空状态检查

### 4. 文档或目录调整
至少做：
- 链接检查
- 目录归属检查
- 命名一致性检查
- 模板或结构一致性检查

---

## 无法验证时的处理
若环境限制导致无法完整验证，必须说明：
- 哪一步未验证
- 为什么不能验证
- 当前完成的是“修改完成”还是“验证完成”
- 潜在风险在哪里

---

## 不可接受的做法
- 只改代码，不跑任何验证
- 因为本地环境缺失就直接宣称完成
- 只看编译通过就当业务正确
- 跳过明显必要的手动验证
