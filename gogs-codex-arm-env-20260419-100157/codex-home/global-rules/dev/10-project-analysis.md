# 10-project-analysis

## 目的
本文件定义接手开发任务时的分析顺序，避免在未理解项目的情况下直接改代码。

---

## 开始前必须回答的问题
在正式修改前，先尽量弄清楚：
- 项目主要目标是什么
- 当前任务属于哪个模块
- 项目使用什么技术栈
- 应用入口在哪里
- 配置入口在哪里
- 测试与构建命令是什么
- 代码风格和目录风格是什么
- 是否存在项目局部规范文件

---

## 优先阅读的内容
根据项目类型，优先查找：
- `README.md`
- `AGENTS.md` / `AGENT.md` / `CLAUDE.md`
- `package.json` / `pyproject.toml` / `Cargo.toml` / `go.mod`
- `Makefile`
- `scripts/`
- `src/` 或主业务目录
- `tests/`
- 配置文件与环境变量示例

---

## 分析原则

### 1. 先理解目录职责
先判断每个核心目录的职责，而不是只看文件名。

默认需要确认：
- 主程序入口
- 业务逻辑层
- UI / API 层
- 数据层
- 配置层
- 测试层
- 脚本层

### 2. 优先复用现有模式
不要一上来重写或引入新模式。先看项目已有做法：
- 命名方式
- 依赖注入方式
- 状态管理方式
- 错误处理方式
- 测试组织方式
- 构建与运行方式

### 3. 找最短闭环
分析时优先找到：
- 如何启动
- 如何跑通最小功能
- 如何复现问题
- 如何验证修复

---

## 进入改动前的最低标准
在开始正式改动前，至少应知道：
- 改哪几个文件最合理
- 为什么在这些地方改
- 改动可能影响到什么
- 如何验证结果

---

## 禁止行为
- 没看清目录职责就大面积改动
- 未确认入口就随意新增模块
- 用自己熟悉的架构替代项目现有架构
- 在不了解验证方式的前提下宣称“已完成”
