# 配置中心重构设计文档

日期：2026-04-15
范围：`frontend` 配置模块信息架构、路由、页面骨架、模块落位
状态：待评审

## 1. 背景

当前前端配置能力集中承载在 `frontend/src/views/ConfigView.vue` 中，单页内同时包含：

- 运行参数
- PLC 映射
- 系统概览与维护
- 数据管理
- 用户管理
- 算法配置
- 物料类型
- 称重设备
- 盘存周期
- 坐标校准

这类“超级标签页”结构已经不再适合继续迭代，主要问题是：

- 信息架构失衡：业务对象、技术对象、维护动作混在同一层级
- 用户路径混乱：现场运维与实施工程师共享同一入口，缺少角色优先级
- 高风险动作暴露方式粗糙：维护操作与普通配置并列，缺少结构隔离
- 页面职责过重：一个视图同时承担导航、状态汇总、编辑、流程任务、危险操作
- 长期不可扩展：继续新增配置项只会让单页继续膨胀

## 2. 目标

本次重构目标不是局部美化，而是将“配置模块”重建为独立子系统。

目标如下：

- 让配置中心优先服务现场运维和管理员
- 让用户先按任务域找入口，再进入具体技术对象
- 将高风险与高级能力结构隔离，但不隐藏
- 让复杂配置页共享统一页面骨架和交互规则
- 尽量复用现有业务能力和 API，不做无意义重写

## 3. 非目标

本次不包含以下内容：

- 不重写后端配置接口契约
- 不全面重写所有配置子模块内部业务逻辑
- 不把整站视觉系统全部推倒重做
- 不把配置中心改造成仅面向研发的工程控制台

## 4. 用户与角色优先级

### 4.1 一等公民

- 现场运维
- 管理员

### 4.2 次级入口

- 实施工程师
- 研发 / 专家用户

设计原则：

- 默认主路径围绕运维任务展开
- 高复杂度技术项允许存在，但要下沉到更明确的域和页面
- 高风险动作不走隐藏式专家模式，而是单独进入“高级与维护”域

## 5. 设计原则

### 5.1 顶层按任务域组织，二级进入技术对象

顶层不再直接展示 PLC、算法、称重、校准这些工程对象，而是先按用户任务域组织。

### 5.2 查看状态与编辑配置分层

域首页先展示态势、异常、待处理项，具体表单进入二级页。

### 5.3 风险动作结构隔离

备份、重启、清理、专家参数与普通运维配置分开承载。

### 5.4 场景语言优先于参数语言

页面应优先使用“采集输入”“自动策略”“设备接入”“业务基础”这类语言，而不是直接暴露碎片化参数。

### 5.5 所有配置都要说明生效方式与影响范围

每个配置页都需要明确：

- 即时生效 / 保存后生效 / 需重启 / 需现场联调
- 影响对象
- 推荐验证动作

## 6. 新的信息架构

新的配置中心拆成独立多路由子系统：

- `/config`
- `/config/operations`
- `/config/devices`
- `/config/business`
- `/config/advanced`

### 6.1 `/config`

配置中心总入口，不再承载巨量表单，只负责：

- 配置域导航
- 状态摘要
- 待处理事项
- 最近变更
- 风险提醒

### 6.2 `/config/operations`

运行与策略中心，面向现场运维和管理员。

子页面：

- `/config/operations/runtime`
- `/config/operations/schedules`
- `/config/operations/algorithm-presets`
- `/config/operations/change-log`

职责：

- 日常运行参数调整
- 自动策略维护
- 算法方案切换
- 配置变更可追踪

### 6.3 `/config/devices`

设备与链路中心，面向管理员和实施。

子页面：

- `/config/devices/overview`
- `/config/devices/plc`
- `/config/devices/scale`
- `/config/devices/camera-lidar`
- `/config/devices/diagnostics`

职责：

- 设备接入配置
- 寄存器与协议映射
- 设备状态与链路验证
- 接入异常排查

### 6.4 `/config/business`

业务基础中心，面向管理员。

子页面：

- `/config/business/materials`
- `/config/business/piles-zones`
- `/config/business/calibration`
- `/config/business/rules`

职责：

- 业务主数据维护
- 料堆/区域基线维护
- 坐标与空间模型维护
- 业务规则承载

### 6.5 `/config/advanced`

高级与维护中心，面向管理员与超管。

子页面：

- `/config/advanced/system`
- `/config/advanced/maintenance`
- `/config/advanced/users`
- `/config/advanced/expert-params`

职责：

- 系统概览与健康状态
- 高风险维护动作
- 用户与权限
- 高级参数与专家配置

## 7. 二级页面定义

### 7.1 运行与策略中心

#### 运行参数总览

目标：

- 替代当前按技术分类堆叠的运行参数大表单
- 改为按运维场景组织

推荐分组：

- 采集输入
- 识别与计算
- 结果输出
- 录像与留档
- 运行保护

行为规则：

- 先看到分组摘要、风险、生效方式
- 再进入具体参数编辑
- 支持页面内保存和统一保存

#### 自动策略

目标：

- 集中承载盘存周期、自动任务、执行窗口

行为规则：

- 用“策略卡 + 编辑面板”的方式代替纯表格堆叠
- 明确显示资源占用与执行影响

#### 算法预设

目标：

- 先管理预设方案，再下钻细节参数

行为规则：

- 运维层看到“场景预设”
- 专家层再进入底层参数

#### 配置变更记录

目标：

- 解决“谁改了什么、是否生效、是否需重启”的追踪问题

### 7.2 设备与链路中心

#### 设备总览

目标：

- 汇总 PLC、称重、相机、雷达的连接状态、配置完成度和最近告警

#### PLC 与寄存器映射

目标：

- 保留现有 `PlcMappingConfig` 能力
- 提升为完整独立工作台

新增重点：

- 地址冲突检查
- 读写权限视图
- 映射测试与连通验证

#### 称重设备

目标：

- 保留现有 `ScaleConfig`
- 重构为“连接参数 + 设备状态 + 测试动作”三段式布局

#### 视觉与雷达接入

目标：

- 把相机/雷达接入基线从运行参数页中拆出
- 集中承载相机码流、Tanway SDK 接入、网络端口配置

#### 链路诊断

目标：

- 集中展示端口、驱动、协议、服务可用性
- 成为排障页，而不是散落在每个页面的提示信息

### 7.3 业务基础中心

#### 物料与密度基线

目标：

- 基于 `DataManageConfig` 升级为业务主数据页

#### 料堆与区域模型

目标：

- 集中管理料堆、区域、关联属性
- 不再仅作为其他页面的数据来源

#### 坐标校准

目标：

- 将当前校准逻辑升级为独立流程任务页

原因：

- 校准本质是多步骤任务，不应继续承载在普通标签页内

#### 业务规则

目标：

- 预留承载体积换算、报表口径、默认业务策略等规则

### 7.4 高级与维护中心

#### 系统概览

目标：

- 只承载版本、依赖、健康状态、配置摘要
- 不混编辑表单

#### 维护操作

目标：

- 统一放置备份、清理、重启等危险动作

#### 用户与权限

目标：

- 将 `UserManagementView` 从超级配置页中彻底拆出

#### 专家参数

目标：

- 承载不适合暴露给普通运维的高级参数
- 成为运维页的下沉入口，而不是主导航一等项

## 8. 页面骨架规范

整个配置子系统统一采用 4 种页面骨架。

### 8.1 域首页骨架

适用于：

- `/config/operations`
- `/config/devices`
- `/config/business`
- `/config/advanced`

结构：

- 域级 `PageHeader`
- 任务卡片区
- 待处理事项 / 最近变更 / 风险提示

要求：

- 不在域首页承载大量表单
- 只做导航、摘要和上下文建立

### 8.2 工作台骨架

适用于：

- 运行参数
- 设备接入
- 物料主数据

结构：

- 顶部标题和主操作
- 第二层过滤 / 分组条
- 左侧对象或分组导航
- 中央编辑区
- 右侧说明 / 风险 / 生效方式 / 影响范围
- 底部悬浮保存条

### 8.3 流程任务骨架

适用于：

- 坐标校准
- 链路诊断中的向导式任务

结构：

- 任务说明
- 步骤导航
- 当前步骤内容
- 固定任务状态卡

要求：

- 必须支持回退、重试、终止
- 不使用普通 tab 模式承载多步骤流程

### 8.4 风险维护骨架

适用于：

- 维护操作
- 专家参数

结构：

- 顶部能力状态
- 中部按风险等级分区
- 每个动作卡显示作用对象、影响说明、前置检查、执行后影响

## 9. 统一交互规则

- 查看态与编辑态分层
- 先场景分组，再展示参数
- 所有配置页都要标记生效方式
- 所有配置页都要显示影响范围
- 高风险动作必须进入危险区
- 复杂配置页必须提供测试 / 校验 / 诊断入口
- 页面内保存与全局保存逻辑必须清晰，不允许用户不知当前改动是否已落库

## 10. 现有模块落位方案

### 10.1 拆除超级容器

`frontend/src/views/ConfigView.vue` 不再作为最终配置承载页，后续将拆为：

- 配置中心壳层页
- 一级域入口页
- 各二级配置页

### 10.2 现有模块去向

- `AlgorithmConfig.vue`
  - 保留业务能力
  - 拆为“算法预设页入口 + 专家参数下沉能力”
- `PlcMappingConfig.vue`
  - 迁入 `/config/devices/plc`
- `ScaleConfig.vue`
  - 迁入 `/config/devices/scale`
- `InventoryScheduleConfig.vue`
  - 迁入 `/config/operations/schedules`
- `DataManageConfig.vue`
  - 迁入 `/config/business/materials`
- `UserManagementView.vue`
  - 迁入 `/config/advanced/users`
- 当前坐标校准逻辑
  - 迁入 `/config/business/calibration`
- 当前系统概览与维护能力
  - 拆到 `/config/advanced/system` 与 `/config/advanced/maintenance`

## 11. 路由与导航策略

### 11.1 路由策略

使用多路由拆分，而不是继续单路由内堆叠 `tab`。

原因：

- 更利于长期扩展
- 更利于面包屑与权限控制
- 更适合保存每页独立状态
- 更容易形成稳定页面职责边界

### 11.2 导航策略

配置中心导航采用三级结构：

- 全局导航：配置中心入口
- 域导航：运行与策略 / 设备与链路 / 业务基础 / 高级与维护
- 页级导航：当前域下二级页面

禁止继续使用“一个左侧按钮列表 + 一个大 tab 容器”承载全部配置能力。

## 12. 权限策略

基础原则：

- 现场运维优先看到运行与策略
- 管理员可见全部域
- 高风险动作和专家参数需要更高权限

建议粒度：

- 域级可见权限
- 页级可编辑权限
- 危险动作可执行权限

## 13. 实施边界与阶段建议

本次实现建议分阶段推进。

### 阶段 1

- 新建配置中心壳层
- 拆分新路由
- 建立 4 个一级域入口页

### 阶段 2

- 优先迁移高价值页面：
  - 运行参数总览
  - PLC
  - 称重设备
  - 物料与密度基线
  - 用户与权限

### 阶段 3

- 迁移校准、链路诊断、专家参数
- 移除旧 `ConfigView` 超级结构

## 14. 风险与注意事项

- 旧模块间存在较强状态耦合，拆页时要避免复用旧容器隐式逻辑
- 运行参数和算法参数边界需要在实现阶段再做一次细化
- 坐标校准与设备链路诊断适合继续拆组件，不宜原样搬迁
- 如后端缺少变更记录能力，`change-log` 页第一阶段仅提供最近配置保存摘要，不提供完整审计历史；完整审计能力需单独补齐后端支持

## 15. 最终结论

本次配置中心重构采用：

- 顶层按任务域组织
- 二级进入技术对象
- 多路由子系统
- 高级与维护独立成域
- 页面骨架统一化

这不是对旧 `ConfigView` 的修补，而是一次配置子系统重建。
