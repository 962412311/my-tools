# 配置中心工业化重构设计文档

日期：2026-04-16
范围：`frontend/src/views/config/**`、配置中心共享组件、配置中心路由与页面职责
状态：已定稿，进入实施

## 1. 背景

当前配置中心已经完成了一轮从单页 `ConfigView.vue` 向多路由结构的迁移，但整体仍处于“架构改了一半”的状态：

- 路由已经拆分为多域结构，但很多二级页只是旧组件外包一层壳
- 一级域首页比二级工作页更完整，造成“入口像产品，落页像临时页”
- 多个页面仍保留占位、过渡文案或明显未完成的实现
- 配置项展示方式仍以参数直出为主，没有形成工业软件所需的操作上下文
- 风险动作、日常配置、业务基线、工程专家参数虽然被分域，但页面内还没有完成真正隔离

用户目标不是继续做一版“好看一些的后台页”，而是把配置中心建设成成熟工业软件的配置工作台。这里的成熟，意味着：

- 入口稳定，用户能按任务而不是按源码模块找功能
- 页面分层清楚，先看状态，再看范围，再做修改
- 危险操作和普通配置有清晰边界
- 页面不再出现明显半成品、占位残留、职责摇摆和临时过渡痕迹
- 新功能未来有固定落点，不会重新堆回大表单

## 2. 目标

本次重构目标：

- 将配置中心整理为完整可扩展的工业软件配置子系统
- 保留现有后端接口契约和核心配置能力，重点重做前端结构与布局模型
- 消除占位页、临时页和“旧组件硬迁移”的割裂感
- 统一所有配置页的布局骨架、信息密度、风险表达和动作位置
- 让运维、管理员、实施和专家用户都能沿着明确的任务路径工作

## 3. 非目标

本次不做：

- 不改后端 API 契约
- 不重写权限模型
- 不系统重做全站视觉语言
- 不为了拆分而重写所有已有功能组件内部逻辑
- 不把配置中心做成只面向研发的工程控制台

## 4. 现状诊断

### 4.1 已完成部分

- `/config` 顶层已经升级为域入口，不再承载全部表单
- `operations / devices / business / advanced` 四个主域已建立
- 共享组件 `ConfigPageShell`、`ConfigDomainHero`、`ConfigQuickStatus` 已具备雏形
- 部分旧模块已迁移为独立路由页

### 4.2 明显未完成部分

#### A. 页面成熟度不一致

- `OperationsHomeView`、`DevicesHomeView` 等域首页已经具有产品化结构
- 但 `ScheduleStrategyView`、`PlcConfigView`、`MaterialsView` 等页仍是“直接塞旧组件”
- `BusinessRulesView` 仍是纯占位页
- `CalibrationView` 仍带有“后续再拆”的过渡描述
- `PilesZonesView` 只有单表格，未形成工作台布局

#### B. 页面骨架未统一

不同页面对以下结构没有统一标准：

- 页面摘要区
- 配置对象分组区
- 前置检查/生效说明区
- 操作按钮区
- 危险动作区
- 数据表 / 表单 / 说明的层次顺序

#### C. 从“参数页”到“工作台页”的转型未完成

工业软件配置页不能只是参数列表，还需要回答：

- 这个页面负责什么
- 修改前先看什么状态
- 生效方式是什么
- 影响范围是什么
- 修改后如何验证
- 是否存在高风险动作

目前多数二级页仍没有承载这些信息。

#### D. 旧大页遗留逻辑仍在影响新结构

`ConfigView.vue` 仍保留大量旧逻辑与旧页面组织方式，说明新结构虽然上线，但尚未完成职责切断。只要这个旧大页仍然是“完整备用方案”，新结构就容易继续漂移。

## 5. 设计原则

### 5.1 配置中心是“工作台”，不是“参数仓库”

页面首先要围绕任务和结果组织，再承载参数。

### 5.2 统一采用“三层布局”

每个成熟配置页都必须遵循统一层次：

1. 状态与上下文层
2. 任务分区与编辑层
3. 验证与风险层

### 5.3 所有页面都显式说明生效方式

至少说明以下一种或多种：

- 即时生效
- 保存后生效
- 需重启
- 需联调验证
- 影响后续任务，不影响历史数据

### 5.4 高风险动作永远独立成区

重启、清理、批量覆盖、校准、专家参数等不能和普通编辑表单混在同一视觉层。

### 5.5 域首页负责分发，专题页负责执行

域首页只做：

- 状态摘要
- 入口分发
- 风险提示
- 任务优先级说明

具体编辑动作在专题页完成，不再让域首页兼做大表单。

### 5.6 以扩展落点为导向

这次不仅要修当前页面，还要为后续新增配置预留稳定落点，避免功能再次回流到 `Runtime` 或 `Expert` 两个兜底页。

## 6. 最终信息架构

保留四大主域，但重新定义各页职责。

### 6.1 `/config`

角色：配置中心总入口

内容：

- 四大主域入口卡
- 全局摘要卡
- 待关注项
- 最近变更
- 风险提醒

要求：

- 不放具体编辑表单
- 让用户在 5 秒内判断“去哪个域”

### 6.2 `/config/operations`

角色：运行与策略域

子页：

- `/config/operations`
- `/config/operations/runtime`
- `/config/operations/schedule`
- `/config/operations/algorithm`
- `/config/operations/change-log`

页面职责：

- `operations home`：承接日常运维入口分发
- `runtime`：按运行场景组织参数，解决“运行时怎么调”
- `schedule`：处理盘存策略与执行窗口，解决“什么时候执行”
- `algorithm`：处理算法预设与必要的下钻入口，解决“按什么方案执行”
- `change-log`：处理变更回溯，解决“谁改过什么”

### 6.3 `/config/devices`

角色：设备与链路域

子页：

- `/config/devices`
- `/config/devices/plc`
- `/config/devices/scales`
- `/config/devices/access`
- `/config/devices/diagnostics`

页面职责：

- `devices home`：先看接入状态和异常分布
- `plc`：寄存器映射工作台
- `scales`：称重设备接入与状态
- `access`：相机、录像、雷达等接入基线
- `diagnostics`：链路、端口、驱动、服务自检

### 6.4 `/config/business`

角色：业务基线域

子页：

- `/config/business`
- `/config/business/materials`
- `/config/business/piles`
- `/config/business/calibration`
- `/config/business/rules`

页面职责：

- `business home`：业务主数据与模型入口总览
- `materials`：物料与密度基线
- `piles`：料堆与区域模型
- `calibration`：校准前提、步骤和参数
- `rules`：业务规则配置落点

### 6.5 `/config/advanced`

角色：高级维护域

子页：

- `/config/advanced`
- `/config/advanced/maintenance`
- `/config/advanced/users`
- `/config/advanced/expert`

页面职责：

- `advanced home`：系统概览与高风险入口分发
- `maintenance`：危险动作工作台
- `users`：账号与权限管理
- `expert`：工程级高级参数

## 7. 统一页面布局模型

所有专题页统一为以下结构：

### 7.1 顶部页头

包含：

- 页面标题
- 一句话职责说明
- 页面级主动作

### 7.2 摘要行

使用状态卡片表达：

- 当前对象数量
- 风险项数量
- 即时生效/需重启/需验证数量
- 当前健康状态

### 7.3 操作前说明区

每页必须明确：

- 这个页面改什么
- 修改会影响谁
- 修改后如何验证
- 哪些内容需要谨慎处理

### 7.4 主工作区

由以下两类结构组成：

- 分区卡片
- 表格/表单工作区

不再使用“裸表格 + 裸表单”直出。

### 7.5 风险与验证区

页面底部或侧边必须保留：

- 生效方式
- 推荐验证动作
- 高风险动作隔离区

## 8. 各页面的成熟化设计

### 8.1 运行参数页

现状问题：

- 已有基础分组能力，但仍接近“参数卡片集合”
- 缺少页面级说明、分组说明、验证建议、批量保存上下文

目标布局：

- 顶部摘要：总参数、即时生效、需重启、当前分组
- 左侧或顶部分组导航：采集输入、识别计算、结果输出、录像留档、运行保护
- 中央参数工作区：按分组显示参数卡
- 底部验证区：说明保存后如何检查

决定：

- 保留动态 schema 驱动
- 不在这轮重写字段组件体系
- 重点补齐页面上下文与分区布局

### 8.2 自动策略页

现状问题：

- 只是把 `InventoryScheduleConfig` 包了一层壳
- 缺少策略摘要、资源影响、启停窗口解释、目标对象范围

目标布局：

- 摘要卡：启用策略数、涉及料堆数、执行窗口、异常配置数
- 说明区：策略作用方式与冲突风险
- 主区：现有策略组件
- 辅助区：策略验证建议与修改后检查项

### 8.3 算法预设页

现状问题：

- 功能可用，但依然更像“内部配置跳板”
- 缺少预设层和高级参数层的边界描述

目标布局：

- 顶部摘要：预设数量、关联参数分组、是否包含高风险参数
- 主区分两层：
  - 预设选择与说明
  - 进入运行参数的下钻入口
- 明确告诉用户哪些变更属于普通策略，哪些属于工程级调参

### 8.4 配置变更记录页

现状问题：

- 有表格，但没有摘要、筛选导向和结果语义

目标布局：

- 顶部摘要：最近修改次数、成功/失败、涉及模块
- 表格保留
- 增加页面说明：适用于交接、审计、回溯

### 8.5 PLC 页

现状问题：

- `PlcMappingConfig` 被直接搬进新页
- 页面缺少地址规划、写入风险、验证动作等说明层

目标布局：

- 摘要卡：映射条目、可写项、危险项、连通状态
- 说明区：地址冲突、联调要求、回写风险
- 主区：现有 PLC 组件
- 风险区：修改后必须执行的检查

### 8.6 称重设备页

现状问题：

- 只有旧组件，没有“设备清单 vs 实时状态 vs 参数配置”的层次

目标布局：

- 摘要卡：在线状态、设备数量、异常数
- 说明区：称重链路影响范围
- 主区：现有配置组件
- 验证区：重量采样/状态验证说明

### 8.7 视觉与雷达接入页

现状问题：

- 当前是 schema 过滤后直接输出
- 接入类配置与运行参数页内部展示逻辑非常相似，重复度高

目标布局：

- 摘要卡：接入项数、录像项数、需重启项数
- 分区：
  - 相机接入
  - 录像留档
  - 网络/链路基线
- 保留 schema 驱动，但按分区渲染，而不是平铺

### 8.8 链路诊断页

目标定位：

- 作为设备域的“先检查、再修改”入口

目标布局：

- 系统服务
- 端口与链路
- 驱动与依赖
- 最近异常与建议动作

### 8.9 物料页

现状问题：

- 只是旧主数据组件的简单承载

目标布局：

- 摘要卡：物料数量、启用状态、密度基线完整度
- 主区：现有数据管理组件
- 说明区：影响重量换算与报表口径

### 8.10 料堆与区域页

现状问题：

- 只有一个列表，远未达到工作台级页面

目标布局：

- 摘要卡：料堆数、区域数、最近更新
- 主区分为：
  - 料堆表格
  - 区域归属与模型关系说明
- 如果后端能力不足，不虚构编辑功能；这轮先把“查看结构”做成成熟页

### 8.11 校准页

现状问题：

- 页面自己承认是过渡态
- 只显示参数，没有流程信息

目标布局：

- 顶部摘要：校准参数数量、最近状态、影响范围
- 流程说明区：
  - 校准前提
  - 操作步骤
  - 保存后验证
- 参数区：保留现有可编辑项
- 风险区：显式标识这是影响空间定位的高风险项

决定：

- 这轮不虚构完整校准向导
- 但必须把校准页从“参数列表”升级为“流程+参数”页

### 8.12 业务规则页

现状问题：

- 纯占位，必须消除

目标布局：

- 先构建“规则总览页”而不是假装已有完整规则引擎
- 展示当前纳入该域的规则来源、未来承载范围和已有配置入口
- 结合现有运行/业务配置中可识别的规则项，先做规则分类与落点说明

决定：

- 这轮把它做成真正的“规则工作台入口页”
- 不再保留单段占位文案

### 8.13 维护页

现状问题：

- 已具备雏形，但危险动作的前置检查和结果反馈还不够强

目标布局：

- 摘要卡：备份目录、systemctl 状态、mysqldump 状态
- 主区：动作卡片
- 每张卡片必须明确：
  - 影响范围
  - 是否中断服务
  - 建议执行时机

### 8.14 用户与权限页

现状问题：

- 页面本身成熟度高于其他页，但仍未完全接入配置中心上下文

目标布局：

- 外层补一个配置域上下文说明区
- 保持原本成熟的用户工作台结构
- 不重复造表格逻辑

### 8.15 专家参数页

现状问题：

- 直接平铺 schema 字段，缺少风险分组和解释

目标布局：

- 摘要卡：参数数量、分组数量、高影响项数量
- 分区渲染：按 group 聚合而不是完全平铺
- 说明区：哪些参数只适合联调或研发支持场景

## 9. 共享组件重构策略

本次重构不靠在每个页面重复写布局，而是增强共享骨架。

### 9.1 `ConfigPageShell`

升级为专题页标准骨架，支持：

- 顶部说明
- actions 插槽
- 摘要区插槽
- 说明区插槽
- 风险区插槽

### 9.2 新增页面分区组件

建议新增：

- `ConfigSectionBlock`
  - 用于承载页面中的独立功能分区
- `ConfigInfoPanel`
  - 用于展示生效方式、影响范围、验证建议
- `ConfigActionCard`
  - 用于维护动作或高风险动作卡片

这些组件服务的是布局一致性，不承载业务逻辑。

### 9.3 `ConfigDomainLayout`

目标：

- 继续负责域内导航
- 去掉无意义 placeholder 心智
- 更像稳定的左侧域导航 + 右侧专题工作区

决定：

- 移除 `Component` 为空时的临时 placeholder 分支
- 域布局只服务真实页面

## 10. 与旧 `ConfigView.vue` 的关系

目标是完成职责切断，而不是长期并存。

策略：

- 保留文件短期存在，避免在本轮牵动太多旧依赖
- 配置中心实际入口全面以新路由为准
- 不再继续给 `ConfigView.vue` 增加任何新能力
- 当新结构主要页面成熟后，将旧页降为遗留兼容或删除候选

## 12. 验收标准

完成后，配置中心应满足：

- 不存在明显占位页或“后续再拆”文案
- 每个二级页都有完整的工作台结构，不再是简单包壳
- 高风险动作、普通编辑、流程任务、业务基线各自有稳定落点
- 页面都能回答“改什么、影响谁、何时生效、怎么验证”
- 后续新增配置可以自然落到对应域，不需要重新堆回旧大页

## 13. 本次决策摘要

- 保留四域结构，不再重新发明顶层导航
- 全面重做页面布局职责，而不是只补 UI
- 保留后端接口与核心旧组件，优先重构承载方式
- 先让配置中心成为成熟工业软件工作台，再考虑细节视觉打磨
