# 点云显示与融合全链路梳理

> 文档定位说明：本文档用于解释当前点云显示、融合和历史消费链路的实现语义，适合作为联调和源码阅读参考；现行 WebSocket 消息字段、部署入口和验收口径请以 `docs/system-architecture.md`、`docs/gogs-rtk.md` 和对应源码为准。

## 1. 目标与范围

这份文档从“当前代码真实怎么跑”出发，把点云相关的几条链路完整拆开：

- 实时监控页单帧点云显示链
- 后端点云处理、轨迹窗口融合、全局表面图更新链
- 全局地图实时显示链
- 库存快照导出、点云文件下载、回放页加载链
- 历史轨迹回放链

这份文档不重复展开算法设计推导，算法原理仍以这两份文档为主：

- `DOC/点云现场算法总览README.md`
- `DOC/点云表面层轨迹融合设计说明.md`

这里更关注“数据对象是什么、由谁产生、经过哪些处理、最终被谁消费”。

## 2. 先分清 5 类点云/轨迹对象

当前系统里最容易混淆的，不是某个函数细节，而是下面 5 类对象语义不同：

### 2.1 原始单帧点云

- 来源：`LidarDriverSdk::OnPointCloud()`
- 数据内容：Tanway SDK 当前回调的一帧点云
- 坐标语义：雷达本地坐标系
- 当前用途：
  - 直接推给 WebSocket 监控页单帧流
  - 送入 `FusionSegmentService` 做段式融合采集

### 2.2 段式融合结果

- 来源：`FusionSegmentService`
- 处理顺序：
  - 读取 PLC 最新位姿
  - 雷达外参变换
  - PLC 位姿变换到世界坐标
  - 直接量化到段级 canonical surface cells
- 坐标语义：世界坐标系
- 当前用途：
  - 轨迹段结束/停稳后提交一次 `fusion preview`
  - 输出 `measurement cloud`
  - 产出融合诊断

### 2.3 旧逐帧分析后帧（兼容链）

- 来源：`PointCloudProcessor::commitFrameResult()`
- 处理顺序：
  - 时间边界细化
  - ROI 裁剪
  - SOR 统计离群点过滤
  - 体素降采样
  - 轨迹窗口聚合
- 坐标语义：世界坐标系
- 当前用途：
  - 保留旧处理器兼容能力
  - 不再作为当前融合主链的默认入口

### 2.4 全局地图快照

- 来源：`GlobalMapManager`
- 数据内容：二维表面单元重建出的全局 `PointCloudPtr`
- 坐标语义：世界坐标系
- 当前用途：
  - WebSocket `globalMap`
  - 盘存测量
  - 库存快照导出 `PCD`
  - 回放页点云加载

### 2.5 回放轨迹

- 来源：`MaterialHistoryManager::queryMonitorTrajectory()`
- 数据内容：`material_operations` 表中的 `gantry_position_x/y`
- 坐标语义：业务历史中的行车轨迹点，`z` 固定为 `0`
- 当前用途：回放页轨迹叠加线

它不是 `PointCloudProcessor` 当前运行中的短轨迹窗口，也不是实时监控页点云轨迹。

## 3. 启动时的主接线

主接线在 `backend/src/core/Application.cpp`，当前关键事实如下：

- 应用启动时创建 `PointCloudProcessor`
- 应用启动时还会创建独立的 `FusionSegmentService`
- `ConfigManager` 中的大部分处理参数在这里灌入处理器
- `fusion/*` 相关配置会灌入 `FusionSegmentService`
- `enableVolumeCalculation(false)` 当前默认关闭持续体积计算
- `LidarDriver::pointCloudReceived` 不再直接接到 `PointCloudProcessor::processFrame`
- `LidarDriver::pointCloudReceived` 当前主接线改为 queued connection 到 `FusionSegmentService::processFrame`
- `WebSocketServer` 同时拿到：
  - `PointCloudProcessor *`
  - `LidarDriver *`
  - `ModbusClient *`
  - `ConfigManager *`
- `PointCloudProcessor` 线程仍会启动，但主要保留兼容对象生命周期、旧信号总线和旧 API 入口，不再承担当前默认融合主路径

这意味着当前系统从结构上天然分成两条并行链：

- 一条是“原始单帧 -> WebSocket -> 监控页”
- 一条是“原始单帧 -> FusionSegmentService -> fusion preview / measurement cloud / diagnostics”

后面所有显示语义混淆，基本都来自这两条链被误认为一条。

## 4. 实时监控页单帧点云显示链

### 4.1 后端入口

入口在 `backend/src/drivers/lidar/LidarDriverSdk.cpp`：

- `OnPointCloud()` 收到 SDK 回调
- `convertToPointCloud()` 把 `UserPointCloud` 转成 `pcl::PointCloud<pcl::PointXYZI>`
- 拷贝字段：
  - `x`
  - `y`
  - `z`
  - `intensity`
- 设置 `frame_id`、`stamp`
- 发出 `pointCloudReceived(cloud)`

当前这一步还有一个关键运行时事实：

- 后端启动时会显式调用 SDK `SetAngleRange()`
- `tanway_sdk/launch/fov_enabled=false` 时，不再沿用 SDK 默认 `30~150°`
- 当前默认会按机型使用安全的近全角默认值：
  - `LT_TW360`：`0.01~360.0°`
  - 其它当前接入机型：`0.1~179.9°`
- `tanway_sdk/launch/fov_enabled=true` 时，才按 `tanway_sdk/launch/fov_min_deg` / `tanway_sdk/launch/fov_max_deg` 裁切

这一步没有做：

- PLC 位姿变换
- ROI
- SOR
- adaptive/temporal/ring 过滤
- 地图融合

### 4.2 WebSocket 单帧流

入口在 `backend/src/service/WebSocketServer.cpp`：

- `WebSocketServer` 直接连接 `m_lidar -> pointCloudReceived`
- `onRawPointCloudReceived(PointCloudPtr cloud)` 收到原始单帧
- `prepareFrameDisplayPointCloud()` 做显示压缩

当前单帧显示压缩链已经明确收口为：

1. `copyFinitePoints()` 去掉非有限点
2. 可选读取 `processing/display_frame_directional_clip_*` 做本地坐标六向几何裁切
3. 读取 `processing/voxel_size`
4. 只做一次 `voxelDownsampleForDisplay()`
5. 组装 WebSocket `pointCloud` 消息

当前明确不会做：

- 坐标变换
- adaptive frame boundary
- temporal boundary refinement
- ring visibility filter
- ROI 裁剪
- SOR
- 点数上限裁剪
- 按 `display_frame_target_points` 重新估算更大的显示体素

所以现在监控页单帧流的真实语义是：

- 数据源：Tanway SDK 原始单帧
- 坐标系：雷达本地坐标系
- 默认保留处理：有限点过滤 + `processing/voxel_size` 体素降采样
- 可选附加处理：`processing/display_frame_directional_clip_enabled=true` 时，按 `+X/-X/+Y/-Y/+Z/-Z` 主轴、半角、偏移和取补规则做单帧几何裁切

### 4.3 WebSocket 单帧消息结构

`buildPointCloudPayload()` 当前统一产出这些字段：

- `type`
- `data.source`
- `data.sequence`
- `data.timestamp`
- `data.rawCount`
- `data.displayCount`
- `data.voxelSize`
- `data.points`

其中：

- `type=pointCloud` 对应单帧流
- `source=pointCloud`
- `rawCount` 是输入点数
- `displayCount` 是下采样后点数
- `voxelSize` 是这次显示使用的体素大小

### 4.4 前端 store 消费

前端入口在 `frontend/src/stores/system.js`：

- `handleMessage()` 按 `message.type` 分发
- `pointCloud` 存单帧点数组
- `pointCloudMeta` 存序号、时间戳、点数和体素元数据
- `acceptStreamUpdate()` 只接受严格递增的 `sequence`

当前 `pointCloudMeta` 字段包括：

- `sequence`
- `timestamp`
- `rawCount`
- `displayCount`
- `voxelSize`
- `source`

### 4.5 监控页渲染

前端入口在 `frontend/src/views/MonitorView.vue`：

- `activePointCloudPoints` 按当前模式选择 `systemStore.pointCloud` 或 `systemStore.globalMap`
- `activePointCloudMeta` 同步选择元数据
- `updatePointCloudMesh(points, meta)` 每次重建 Three.js `BufferGeometry`
- 点颜色按 `z` 值着色
- 页面显示 `rawCount / displayCount / voxelSize`

当前还有一个非常重要的前端行为：

- 如果当前首选流为空，`activePointCloudPoints` 会自动回退到另一条流

这意味着在监控页里，用户虽然选的是“单帧”或“地图”，但在某些时刻实际看到的可能是另一条数据链。

## 5. 当前段式融合主链

### 5.1 当前主路径

当前新的默认主路径已经切到段式融合链：

1. `FusionSegmentService::processFrame()` 收到雷达原始帧
2. 读取 `ModbusClient` 最新位姿
3. 把原始点转换到世界坐标
4. 直接更新开放轨迹段的 canonical surface cells
5. 满足停稳/距离/时间/flush 条件时提交一次段结果
6. 同源产出：
   - `fusionPreviewReady`
   - `measurementCloudReady`
   - `fusionDiagnosticsUpdated`

当前段提交触发条件包括：

- `stationary`
- `distance_or_time`
- `trajectory_segment`
- `flush/stop`

### 5.2 旧 `PointCloudProcessor` 的现状

`PointCloudProcessor` 相关的异步队列、`analyzeFrame()`、`commitFrameResult()`、`processFrameInternal()` 仍然保留在仓库中，但截至当前代码：

- 它不再接收 `LidarDriver::pointCloudReceived` 作为默认主入口
- 当前主融合链不再依赖它逐帧分析
- 它更多承担兼容对象、旧 API 和部分信号透传角色

因此后续看代码时必须区分：

- `FusionSegmentService`：当前默认融合主链
- `PointCloudProcessor`：仍在仓库中的旧逐帧兼容链
- 最近窗口丢帧统计

### 5.2 分析阶段

`PointCloudProcessor::analyzeFrame()` 当前顺序如下：

1. 原始输入记为 `result.originalCloud`
2. 可选执行 `extractAdaptiveFrameBoundary()`
3. 执行雷达外参标定变换
4. 执行 PLC 位姿变换到世界坐标
5. 计算有效可视半径
6. 可选执行 `filterCloudByAnnularVisibility()`

这一阶段输出的关键对象包括：

- `adaptiveCloud`
- `calibratedLocalCloud`
- `globalCloud`
- `ringVisibleCloud`

到这一步为止，点云已经从雷达坐标进入世界坐标。

### 5.3 提交阶段

`PointCloudProcessor::commitFrameResult()` 当前顺序如下：

1. 读取当前丢帧/队列统计
2. 判断是否静止及位姿步进距离
3. 可选执行时间边界细化
4. 生成显示用点云
5. `cropRoi()`
6. `statisticalOutlierFilter()`
7. `voxelDownsample()`
8. 自动标定
9. `appendToTrajectoryWindow()`
10. 更新全局地图
11. 检测最高点并写 PLC
12. 可选做体积估算
13. 发出显示、诊断和地图相关信号

这里要特别区分两种“显示云”：

- 监控页单帧流使用的是 `WebSocketServer` 直接拿到的原始单帧
- `commitFrameResult()` 里的 `displayCloud` 属于处理器内部派生显示对象

当前仓库里没有看到 `pointCloudProcessed` 的实际消费者，因此监控页并不是靠这条信号链拿单帧点云。

### 5.4 处理器内部显示云语义

`commitFrameResult()` 里会根据 `displayFrameDownsampleOnly` 选择显示源：

- `true`：`displaySourceCloud = result.originalCloud`
- `false`：`displaySourceCloud = temporallyFilteredCloud`

然后统一走 `buildDisplayPointCloud()`

当前 `buildDisplayPointCloud()` 已经被收口为：

- 只做体素降采样
- 使用 `voxelSize`
- 不再按目标点数再次估算叶子尺寸
- 不再叠加其他显示裁剪逻辑

但这条链当前主要影响处理器内部信号，不是 WebSocket 单帧监控流。

### 5.5 ROI、SOR、体素降采样后的主产物

`commitFrameResult()` 真正用于业务主链的，是 `filteredCloud`：

- 输入：`temporallyFilteredCloud`
- 再经过：
  - ROI
  - SOR
  - voxelDownsample

这个 `filteredCloud` 后续被用于：

- 轨迹窗口聚合
- 地图更新
- 最高点
- 体积
- 诊断

因此它才是“融合建图主链”的核心单帧对象，而不是监控页看到的原始单帧显示云。

## 6. 轨迹窗口融合与全局表面图

### 6.1 轨迹窗口触发逻辑

`PointCloudProcessor::appendToTrajectoryWindow()` 当前按移动状态工作：

- 静止且已有活动窗口：
  - 先把当前帧并入窗口
  - 立刻 flush
  - 原因通常记为 `stopped`
- 静止且没有活动窗口：
  - 当前帧直接返回
  - 原因记为 `stationary`
- 移动且没有活动窗口：
  - 创建新窗口
  - 原因记为 `accumulating`
- 移动过程中方向变化达到阈值：
  - 提前 flush
  - 原因记为 `direction_change`
- 移动过程中累计距离、时长、帧数达到阈值：
  - flush
  - 原因记为 `distance_or_time`

当前轨迹窗口消耗的是 `filteredCloud`，不是原始点云。

### 6.2 轨迹窗口统计

处理器会记录这些轨迹相关诊断：

- `trajectoryFlushReason`
- `lastTrajectoryFlushReason`
- `lastTrajectoryFlushFrames`
- `lastTrajectoryFlushDistance`
- `trajectoryChunkFrames`
- `trajectoryChunkDistance`
- 累计的各类 flush 次数

这套字段才是“当前后端真实轨迹融合行为”的诊断依据。

### 6.3 GlobalMapManager 的真实职责

`backend/src/processing/pcl/GlobalMapManager.cpp` 当前做的是二维表面图，不是三维体素地图。

每个表面单元当前维护：

- `centerX`
- `centerY`
- `zMean`
- `zTop`
- `candidateTop`
- `candidateHits`
- `intensityMean`
- `m2`
- `hitCount`
- `stableHitCount`
- `confidence`
- `lastUpdateTs`

地图更新方式是：

- 按世界坐标 `x/y` 落单元
- 增量融合表面统计
- 重建全局 `PointCloudPtr`
- 发出 `globalMapUpdated`

所以当前全局地图的真实语义是：

- 来源：处理后轨迹块/单帧
- 坐标：世界坐标
- 形态：二维表面层重建出的点云快照

## 7. 实时全局地图显示链

### 7.1 后端推送

`WebSocketServer` 当前仍会连接 `PointCloudProcessor::globalMapUpdated`，但这条链路已经从“只推旧全局地图”演变为“保留 `globalMap` 兼容流，同时补推融合预览语义”：

- `onGlobalMapUpdated(PointCloudPtr map)`
- `prepareDisplayPointCloud(map, true)`
- `buildPointCloudPayload("globalMap", ...)`
- 先广播 `type=globalMap`
- 再调用 `onFusionPreviewUpdated(map)`，补发当前融合预览流

也就是说，`globalMapUpdated` 现在更接近“后端内部兼容信号”，前端现行主线需要同时区分：

- `globalMap`：兼容性的全局表面图快照
- `fusionPreview`：当前融合预览显示语义
- `pointCloud`：雷达本地单帧显示语义

### 7.2 地图显示压缩

和单帧流不同，当前 `globalMap` 仍然保留了更激进的显示压缩：

- 先过滤有限点
- 按目标点数估算显示体素
- 体素降采样
- `limitDisplayPointCount()`

因此当前两条实时显示链并不对称：

- `pointCloud`：只保留 `processing/voxel_size`
- `globalMap`：继续按目标点数估算显示体素并限点

### 7.3 前端渲染

监控页前端没有区分两条流的坐标语义，只是把它们都当作 `points[]` 放进同一个 Three.js 视图。

这会带来一个重要事实：

- 单帧流通常是雷达本地坐标
- 全局地图是世界坐标

它们不是天然可直接叠加或无缝回退的同一类数据。

## 8. 库存快照、点云文件下载、回放页链路

### 8.1 库存快照创建

`MaterialHistoryManager::createInventorySnapshot()` 当前会：

1. 汇总当前料堆体积/重量
2. 如果没有外部指定 `pointCloudFile`
3. 调用 `exportSnapshotPointCloudFile(now)`
4. 把结果写入 `inventory_snapshots.point_cloud_file`

### 8.2 快照点云文件来源

`exportSnapshotPointCloudFile()` 当前不是导出“某一帧原始点云”，而是：

- 调用 `m_processor->getGlobalMap()`
- 把当前全局地图写成 `data/pointcloud/snapshots/*.pcd`

因此库存快照点云文件的真实语义是：

- 全局表面图快照
- 不是原始单帧
- 也不是某次轨迹窗口的中间对象

### 8.3 下载链路

`HttpServer` 当前通过 `/api/point-cloud-files/download` 提供下载：

- 做 canonical path 校验
- 只允许 `data/pointcloud/snapshots` 目录下的文件

所以回放页当前能下载和加载的，也是库存快照阶段落盘的全局地图 `PCD`。

### 8.4 回放页点云链

`frontend/src/views/PlaybackView.vue` 当前流程如下：

1. `loadPlaybackMaps()` 调 `/api/inventory-snapshots`
2. `buildMapListFromSnapshots()` 只保留有 `pointCloudFile` 的快照
3. `selectMap()` 选择一个快照
4. `loadPointCloudFromFile(map)` 用 `PCDLoader` 加载下载地址
5. Three.js 场景直接渲染这个 `PCD`

因此回放页点云当前展示的是：

- 某时刻保存下来的全局地图快照文件
- 不是实时监控页单帧流

## 9. 历史轨迹回放链

### 9.1 后端接口

`HistoryApiHandler` 注册了 `/api/monitor/trajectory`

实际数据来源是：

- `MaterialHistoryManager::queryMonitorTrajectory()`
- 查询表：`material_operations`
- 读取字段：
  - `gantry_position_x`
  - `gantry_position_y`
  - `operation_time`
- 输出字段：
  - `x`
  - `y`
  - `z=0`
  - `timestamp`
  - `pileId`
  - `areaName`

### 9.2 前端消费

`PlaybackView.vue` 当前：

- `loadPlaybackTrajectory()` 调 `getMonitorTrajectory()`
- 得到记录数组后写到 `trajectoryPoints`
- `updateTrajectoryOverlay()` 画回放轨迹线

所以回放页轨迹线当前语义是：

- 历史业务作业轨迹
- 不是处理器实时短轨迹窗口
- 不是点云融合时实际 flush 出来的 chunk 轨迹

## 11. 关键配置项

### 11.1 实时单帧显示直接相关

- `processing/voxel_size`
  - 当前监控页单帧流唯一保留的显示压缩参数
- `processing/display_frame_target_points`
  - 当前对监控页单帧流已不再生效
  - 仅作为兼容旧显示链的保留配置

### 11.2 处理器分析与融合相关

- `processing/adaptive_frame_boundary`
- `processing/adaptive_boundary_*`
- `processing/intensity_assisted_boundary`
- `processing/intensity_boundary_*`
- `processing/temporal_boundary_refinement`
- `processing/temporal_boundary_*`
- `processing/ring_visibility_filter_enabled`
- `processing/lidar_blind_cone_deg`
- `processing/lidar_max_visible_radius`
- `processing/voxel_size`
- `processing/blind_zone_*`
- `processing/auto_calibration_*`

### 11.3 需要特别注意的语义

- `processing/display_frame_downsample_only` 这个名字容易让人以为它直接控制监控页单帧流
- 但当前监控页单帧流是 `WebSocketServer` 直接从 `LidarDriver` 取原始帧推送
- 所以它更多描述的是处理器内部派生显示云语义，而不是 WebSocket 单帧显示链唯一开关

## 12. 当前链路的核心结论

当前点云链路不是“一条主线”，而是至少四条并行语义不同的链：

1. 原始单帧监控链
   `SDK 回调 -> WebSocket pointCloud -> MonitorView`
2. 后端处理融合链
   `SDK 回调 -> PointCloudProcessor -> 轨迹窗口 -> GlobalMapManager -> 最高点/体积/诊断`
3. 全局地图显示链
   `GlobalMapManager -> WebSocket globalMap -> MonitorView`
4. 快照回放链
   `GlobalMap -> snapshot PCD -> PlaybackView`

另外还有一条独立的历史轨迹链：

5. 历史作业轨迹链
   `material_operations -> /api/monitor/trajectory -> PlaybackView`

只要把这 5 类对象混在一起，后续对“为什么显示不一致、为什么回放和实时不一样、为什么坐标不对”就一定会判断失真。

## 13. 当前发现的问题

下面这些问题已经同步记录到 `todo.md`，这里先给出链路视角的摘要：

- 处理器里仍保留一份未接线的 `processFrameInternal()` 旧同步全链路，和当前异步 `analyzeFrame() + commitFrameResult()` 双轨并存，维护风险高。
- `pointCloudProcessed` / `pointCloudDisplayReady` 仍保留处理器内部显示信号，但当前监控页单帧流实际走 `WebSocketServer <- LidarDriver`，语义容易误判。
- `MonitorView` 会在单帧流和地图流之间自动回退，但两者坐标系和来源不同，当前 UI 没有把这种差异显式告诉用户。
- `/api/monitor/trajectory` 名字像“监控点云轨迹”，实际返回的是历史作业记录中的行车位置。
- 监控页已有 `trajectoryPoints` / `showTrajectory` 结构，但当前没有真实数据注入，轨迹显示链没有闭环。
- `system.js` 定义了 `resetPointCloudStreams()`，但 `status=running` 时没有主动重置序号；如果后端重启并把 `sequence` 从小值重新开始，前端有概率继续丢新帧。
- `docs/system-architecture.md` 的 WebSocket 协议描述仍停留在旧版，未体现 `sequence/rawCount/displayCount/voxelSize/source` 等真实字段。

## 14. 建议后续梳理顺序

如果后面要继续收口这条链，建议按下面顺序处理：

1. 先把“单帧流 / 地图流 / 快照流 / 历史轨迹流”的命名和文档口径统一
2. 再清理处理器里未接线的旧同步链和无消费者信号
3. 再决定监控页是否允许“不同坐标系流自动回退”
4. 最后再考虑是否补实时轨迹显示或改 `/api/monitor/trajectory` 命名

这样做的原因很简单：当前最大问题不是算法细节不够，而是链路语义太容易被混淆。
