# 视频浏览器兼容矩阵

## 目标

这份文档用于把当前项目的视频浏览器兼容建议固定下来，避免现场在 `H.264 / H.265`、`WebRTC / HLS` 和不同浏览器之间反复试错。

当前约束：

- 这是项目级兼容建议，不等于真实摄像头环境最终验收结论
- 真实海康设备、目标机硬件解码能力、浏览器版本都可能改变最终结果
- 现场最终结论仍需回写到 `todo.md`、`DOC/现场联调验收README.md` 和联调记录模板
- 真实摄像头回写统一使用 [`视频实机验收矩阵模板`](视频实机验收矩阵模板.md)
- `2026-04-19` 已用 headless Chromium 打开真实监控页并确认页面状态卡正常加载，但它没有形成可信的实时画面证据，因此不能直接作为兼容矩阵结论

## 当前默认基线

- 主码流：录像优先，默认 `2K / 15fps~25fps`
- 子码流：实时预览优先，建议 `720p / 15fps`
- 子码流编码：Web 端联调优先建议 `H.264`
- 主码流编码：按录像链路和现场浏览器兼容性在 `H.264 / H.265` 之间确认

## 实测前固定准备动作

先运行：

```bash
rtk bash scripts/arm/verify_browser_matrix_readiness.sh
```

这一步不替代真实浏览器验收，它只负责先固定：

- 当前 `monitor` 页面地址
- 当前 `HLS` 播放地址
- 当前 `WebRTC player` 地址
- 当前主/子码流分辨率、帧率、编码基线
- 当前 `recording/input_codec`
- 当前 `stream-info / self-check / recording-status` 是否已具备浏览器实测前置条件

只有这一步返回 `PASS` 后，才进入真实浏览器矩阵回写。

## 浏览器兼容建议

| 浏览器 | 子码流 H.264 | 子码流 H.265 | 当前建议 |
| --- | --- | --- | --- |
| Safari | 相对稳妥 | 相对更可接受 | 若现场必须保留 H.265，优先在 Safari 上验证 |
| Chrome | 稳妥 | 依赖硬件/环境 | 默认仍建议子码流先走 H.264 |
| Edge | 稳妥 | 依赖硬件/环境 | 与 Chrome 同类处理，优先保留 H.264 降级 |
| Firefox | 稳妥 | 风险高 | H.265 场景优先改回 H.264 |
| 未知浏览器 | 待确认 | 风险高 | 优先按 H.264 子码流联调 |

## 流协议建议

- `WebRTC/WHEP`：实时优先，适合作为默认主线
- `HLS`：作为浏览器兼容性和链路异常时的降级方案
- 如果子码流编码为 `H.265`，应提前确认目标浏览器是否可直放，否则默认准备 `H.264` 降级路径

## 现场推荐顺序

1. 先确认监控页顶部“视频基线”
2. 再看“浏览器兼容”
3. 再看“视频自检”
4. 最后做 `WebRTC / HLS / 截图 / 录像 / 回放` 动作验证

4. 最后执行 `rtk bash scripts/arm/finalize_remaining_acceptance_closure.sh`，统一归档当前验收包并更新 `V1.1` / `todo.md`
