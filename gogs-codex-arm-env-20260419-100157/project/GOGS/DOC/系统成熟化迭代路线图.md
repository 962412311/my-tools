# 系统成熟化迭代路线图

## 目标

这份文档用于把当前项目从“功能主线已打通”推进到“可长期交付、可稳定运维、可现场复制”的成熟系统。

这里的“成熟”不是继续堆功能，而是同时满足以下要求：

1. 关键业务链路稳定可用
2. 设备接入边界清楚、异常可诊断
3. 部署、升级、验收和回归有固定方法
4. 文档、脚本、运行状态和版本口径一致

## 当前判断

当前仓库已经完成的内容主要是：

- 点云处理、库存、回放、监控、认证、配置、远控等主线功能闭环
- `mediamtx + GStreamer + ONVIF` 视频主线
- ARM Linux 原生部署、基础验收脚本和 systemd 模板

当前真正阻碍系统成熟的，不是“再加几个页面”，而是以下四类问题：

1. 现场视频链路还缺少长期稳定性和兼容性收口
2. 海康/PLC/称重等设备侧还缺少标准接入与异常诊断体系
3. 运维、自检、告警、升级、回滚还没有形成固定流程
4. 版本目标、验收出口和回归口径还没有正式固化

## 迭代原则

### 1. 先稳主线，再补增强

优先收口：

- 实时监控
- 录像与回放
- PTZ 与抓图
- PLC 与称重
- ARM 目标机部署与验活

不优先做与现场闭环无直接关系的泛化功能。

### 2. 显示链路和录制链路分开设计

视频部分保持两条主线：

- 实时显示：`RTSP -> MediaMTX -> WebRTC/WHEP` 优先，`HLS` 降级
- 正式录制：后端直拉 RTSP 落盘，不依赖浏览器 `MediaRecorder`

海康 SDK 只作为后端增强能力候选，不替代当前 Web 主显示链路。

### 3. 每个阶段都必须四件套收口

每一轮成熟化迭代都必须同时完成：

1. 实现或配置修正
2. 回归或现场验证
3. 文档更新
4. `todo.md` / 状态口径更新

### 4. 版本必须有明确验收出口

后续不再按“零散 TODO”推进，而按版本推进。每个版本都要有：

- 目标
- 范围
- 非目标
- 验收项
- 回归项
- 风险和前置条件

## 成熟化主线

### 主线 A：视频链路成熟化

目标是让海康摄像机在 Web 端实时显示、抓图、录像、回放四条链路稳定可用。

重点任务：

- 把实时预览默认顺序调整为 `WebRTC/WHEP` 优先、`HLS` 降级
- 固化海康相机编码策略，明确主码流/子码流、分辨率、帧率、编码格式
- 补齐 `H.264/H.265 × 浏览器/终端` 兼容矩阵
- 完成长时预览、断流恢复、相机重启恢复、mediamtx 异常恢复实测
- 补齐录像健康检查、自动恢复和分段录像稳定性验证
- 补齐截图失败回退、录像导出、回放拖拽与下载体验验证

### 主线 B：设备接入成熟化

目标是让海康、PLC、称重、雷达等设备接入方式标准化、可复用、可排障。

重点任务：

- 梳理并固化 RTSP / ONVIF / ISAPI / SDK 的职责边界
- 建立海康设备标准接入模板和联调 SOP
- 建立 PLC、称重、雷达的接入前提、自检项和错误码说明
- 补齐设备异常留痕，避免只返回“失败”
- 明确哪些能力必须实机验证，哪些可以本地回归

### 主线 C：业务闭环成熟化

目标是让操作员可以完成完整业务流程，而不是只能由开发者联调。

重点任务：

- 梳理监控页主流程：看视频、看点云、截图、录像、PTZ、写 PLC
- 梳理回放页主流程：按时间查、联动视频/点云/作业、下载留痕
- 补齐异常流程：视频断流、录像失败、PTZ 不可控、PLC 不可写、点云缺失
- 把“代码可用”升级成“现场操作可用”

### 主线 D：运维交付成熟化

目标是让部署、升级、验活、故障处理不依赖个人记忆。

重点任务：

- 整理配置分层和关键配置校验
- 增加一键自检与运行时健康状态汇总
- 固化服务启动顺序、重启策略、日志归档和磁盘回收策略
- 补齐升级、回滚、备份、恢复、验活 SOP
- 把现场确认的新约束沉淀进文档和脚本

### 主线 E：质量保障成熟化

目标是让回归和现场验收有稳定口径。

重点任务：

- 建立视频专项回归清单
- 建立设备联调清单
- 建立部署验收清单
- 补齐关键后端接口测试和前端关键流程测试
- 建立“脚本检查 + 人工验收单”双轨机制

### 主线 F：版本化管理成熟化

目标是让项目推进方式从“持续堆 TODO”升级为“按版本交付”。

重点任务：

- 定义版本目标和边界
- 固化每个版本的验收口径
- 明确完成、待验证、未开始三种状态
- 禁止把“代码已写”直接当成“版本已完成”

## 版本划分

### V1.1 稳定可用版

目标：

- 把现有主线功能收口到“现场可稳定运行”

范围：

- 视频实时预览、抓图、录像、回放
- PTZ 控制与状态诊断
- ARM 目标机构建、部署、服务启动、基本验活
- 关键页面主流程回归

完成判定：

- 视频链路在真实海康设备环境完成一次完整验收
- ARM 目标机完成一次完整构建、部署、启动、验活
- 关键异常路径具备可读错误信息

### V1.2 生产增强版

目标：

- 把系统从“可运行”提升到“可运维、可恢复、可交接”

范围：

- 一键自检
- 服务健康检查与自动恢复
- 磁盘、录像、日志运维能力
- 升级、回滚、备份、恢复 SOP
- 更完整的回归与验收文档

完成判定：

- 现场问题定位不再依赖开发者口头排查
- 关键服务异常后能自动恢复或明确告警

### V1.3 厂商增强版

目标：

- 在不破坏现有 Web 主链的前提下，补充厂商增强能力

范围：

- 评估并按需接入海康 SDK 或 ISAPI 扩展能力
- 补充设备录像检索、设备端回放下载、事件订阅等厂商能力
- 只在真实业务有收益时接入，不为“技术上能做”而接入

完成判定：

- 厂商增强能力有明确业务收益
- 不引入新的部署和维护灾难

## 当前优先顺序

后续一段时间建议固定按下面顺序推进：

1. `V1.1` 视频链路现场收口
2. `V1.1` ARM 目标机构建部署与验活收口
3. `V1.1` 设备联调模板与错误诊断收口
4. `V1.2` 一键自检、健康检查、自动恢复
5. `V1.2` 运维 SOP、升级回滚、日志与磁盘治理
6. `V1.3` 海康厂商增强能力评估与接入

## 不纳入当前优先范围的事项

以下内容暂不作为当前成熟化优先任务：

- 大范围重构现有前后端架构
- 为未知现场提前做泛兼容平台抽象
- 为展示效果而做非关键视觉改造
- 没有明确业务收益的 SDK 接入
