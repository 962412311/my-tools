# 点云表面层轨迹融合设计说明

> 文档定位说明：本文档主要保留轨迹融合方案的设计动机和处理思路，适合作为算法设计参考；现行消息协议、部署入口和联调口径请以 `docs/system-architecture.md`、`docs/gogs-rtk.md` 与当前实现为准。

## 1. 场景约束

当前抓斗行车场景下，激光雷达具备以下实际特征：

- 单帧点云较稀疏，行车静止时实际采样分辨率约为 `12cm x 6cm`
- 行车移动连续且速度有限，相邻帧在空间上具有明显重叠
- 真正可用的表面信息主要来自连续移动过程中的多帧补密
- 业务目标聚焦于表面层：
  - 最高点检测
  - 表面形态展示
  - 体积估算
  - 局部料堆边界与上包络判断

因此，地图更新不能再按“单帧直接写区域点云”的方式处理，而应改为：

- 先按连续轨迹将多帧融合成短轨迹块
- 再将轨迹块按世界坐标增量更新到区域表面网格

## 2. 总体策略

### 2.1 核心原则

- 地图只维护表面层，不维护三维实体体素体
- 单帧稀疏是允许的，不强求静止状态下的局部致密
- 行车移动过程中，利用短轨迹窗口对多帧观测进行补密
- 区域更新不是整块覆盖，而是对世界坐标落入的表面单元做增量融合

### 2.2 更新链路

当前后端主链按如下顺序执行：

1. 雷达 SDK 获取原始点云
2. 自适应有效边界过滤
3. 外参与 PLC `x/y` 位姿变换到世界坐标
4. 时序边界过滤
5. ROI 裁剪
6. SOR 统计离群点过滤
7. 体素下采样
8. 短轨迹窗口融合
9. 区域表面网格更新
10. 派生最高点、盘存和回放所需数据

## 3. 区域与表面单元

### 3.1 区域划分

- 区域仍按世界坐标 `x/y` 进行划分
- `region_size` 支持自动估算和固定配置
- 目标是保证单次轨迹块覆盖 1 到 4 个区域

### 3.2 表面单元

区域内部不再存储“原始点集合”，而是维护二维表面单元。

默认单元尺寸：

- `6cm x 6cm`

每个表面单元维护：

- `z_mean`
- `z_top`
- `candidate_top`
- `candidate_hits`
- `intensity_mean`
- `hit_count`
- `stable_hit_count`
- `confidence`
- `m2`（用于估算局部方差）
- `last_update_ts`

## 4. 短轨迹窗口

### 4.1 设计目的

单帧稀疏本身不是问题，真正有效的是连续移动路径对同一区域的重复覆盖。

因此，地图更新的基本单元不再是单帧，而是短轨迹块。

### 4.2 窗口触发与结算

短轨迹窗口在以下情况下结算：

- 累积位移达到约 `0.30m`
- 累积时间达到约 `1.2s`
- 窗口内帧数达到上限
- 行车从移动切换到静止
- 行车方向出现明显变化

对应到处理诊断时，结算原因会回显为：

- `direction_change`
- `distance_or_time`
- `stopped`
- `stationary`

其中：

- `direction_change` 表示轨迹块因明显转向而提前切块
- `distance_or_time` 表示轨迹块达到位移或时长阈值
- `stopped` 表示在静止过渡时把最后一帧并入并立即收尾
- `stationary` 表示当前帧没有延续到活动轨迹窗口，直接按弱更新处理
- 空窗口停机时会回显 `stop_empty`，表示没有可入图的尾段，不属于异常

### 4.3 静止处理

- 静止时不强行补密
- 若前面存在活动中的移动窗口，则先将终点帧并入窗口后结算
- 若没有活动窗口，则当前帧按稀疏表面直接弱更新

## 5. 表面单元融合规则

### 5.1 `z_mean`

- 用于平滑表面显示、体积估算和整体趋势
- 只有在新观测落入自适应稳定带时才作为稳定观测更新
- 对轻微低值偏差允许弱更新，但不会粗暴下拉表面

### 5.2 `z_top`

- 用于上包络、最高点和安全边界判断
- 不直接采用单次最大值
- 通过 `candidate_top` 机制提升：
  - 单次略高观测先作为候选
  - 多次重复命中后再提升为正式 `z_top`

### 5.3 噪声处理

噪声点不采用“固定阈值直接删掉”的粗暴方式，而采用：

- 稳定带内直接接纳
- 略高观测进入候选上包络
- 明显异常高点延迟信任
- 轻微偏低点允许弱更新
- 极端偏离点忽略，不立即破坏表面层统计

这样可以兼顾：

- 保留真实尖峰和边缘精度
- 抑制孤立噪声和瞬时高飘点

## 6. 当前实现落点

### 6.1 `PointCloudProcessor`

当前负责：

- 生成世界坐标点云
- 维护短轨迹窗口
- 按移动/静止/方向变化等状态决定何时输出轨迹块
- 将轨迹块提交给 `GlobalMapManager`
- 保持最高点、PLC 回写、体积估算和诊断链路不变

### 6.2 `GlobalMapManager`

当前负责：

- 维护全局表面单元哈希
- 按世界坐标将单元归入区域
- 输出 `z_mean` 形式的全局点云快照
- 为历史、回放、下载和前端显示继续提供兼容的 `PointCloudPtr`

## 7. 适用边界

这套策略适用于：

- 行车移动连续且速度有限
- 单帧稀疏但多帧重叠明显
- 业务关注表面层而非实体内部结构

不适合直接照搬到以下场景：

- 高频剧烈抖动
- 快速姿态变化
- 大范围三维实体重建
- 需要完整保留体素内部密度分布的场景

## 8. 后续联调关注点

实机联调时重点看以下诊断项：

- `mapUpdateMode`
- `trajectoryChunkFrames`
- `trajectoryChunkDistance`
- `trajectoryWindowFrames`
- `trajectoryWindowDistance`
- `surfaceCellCount`
- `regionCount`
- `finalKeepRatio`

如果实际联调发现：

- 停止后表面不稳定
- 方向切换处出现断裂
- `z_top` 过高或过低
- 边界过度收缩

则优先调整：

- 轨迹窗口位移阈值
- 方向变化阈值
- 表面单元尺寸
- 候选高点提升次数
- 稳定带和上包络带的自适应范围
