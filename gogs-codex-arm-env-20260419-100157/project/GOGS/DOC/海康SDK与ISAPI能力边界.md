# 海康 SDK 与 ISAPI 能力边界

## 目标

这份文档用于固定海康设备在当前项目中的能力边界，避免后续把 `RTSP / ONVIF / ISAPI / SDK` 混用成一团。

当前项目前提：

- 主操作界面是浏览器，不是 Windows C/S 客户端
- 实时预览主链已经固定为 `RTSP -> MediaMTX -> WebRTC/WHEP` 优先，`HLS` 降级
- 正式录像主链已经固定为后端直拉 `RTSP` 落盘
- 海康增强能力只允许作为后端补强，不允许替换现有 Web 主播放链

## 一、协议职责边界

| 能力 | 当前主责任 | 当前是否主线 | 说明 |
| --- | --- | --- | --- |
| 实时预览 | `RTSP + MediaMTX + WebRTC/WHEP` | 是 | 浏览器端低延迟主链 |
| 浏览器降级预览 | `HLS` | 是 | `WebRTC` 不可用时降级 |
| PTZ 控制 | `ONVIF` | 是 | 已有接口和状态诊断 |
| 抓图兜底 | `ONVIF GetSnapshotUri` | 是 | 浏览器本地截图失败时使用 |
| 正式录像 | 后端 `RTSP + GStreamer` | 是 | 不依赖浏览器录制 |
| 设备参数读取/补强 | `ISAPI` | 候选 | 适合读取设备信息、设备录像检索等 |
| 厂商专有增强 | 海康 `SDK` | 候选 | 只在 Web 主链不能覆盖时补进后端 |

结论：

- `RTSP` 负责码流来源
- `MediaMTX` 负责浏览器消费
- `ONVIF` 负责云台和快照兜底
- `ISAPI / SDK` 不负责替代浏览器实时播放

## 二、明确不做的事

当前版本明确不做这些方案：

- 不把海康 `Web SDK` 作为默认前端播放器
- 不把海康 `SDK` 直接塞进浏览器链路
- 不为了某一台设备的厂商特性，反向重做整条视频主线
- 不让 `ISAPI` 和 `ONVIF` 在同一能力上长期双轨并存

原因：

- 当前系统是 `Web + ARM Linux` 主线，不是 Windows 专用客户端
- 浏览器实时显示已经有统一主链，不应再引入第二套播放器体系
- 双轨协议会让联调、排障和验收口径失控

## 三、ISAPI 适合补的能力

如果后续要引入 `ISAPI`，优先考虑下面这些能力：

1. 设备基础信息读取
2. 编码参数读取与核对
3. 设备端录像检索
4. 设备端录像下载
5. 厂商特有状态查询

适用原则：

- 用于“补信息”和“补设备侧能力”
- 不用于替代当前 `RTSP / ONVIF / GStreamer` 主链
- 接入后必须明确落在后端，不能把协议细节散到前端

## 四、SDK 适合补的能力

如果后续要引入海康 `SDK`，优先考虑下面这些场景：

1. 某些型号设备的录像检索只能通过厂商能力完成
2. 设备事件订阅或告警订阅仅靠 `ONVIF` 不稳定
3. 设备端历史回放下载需要更强厂商支持
4. 必须读取厂商私有字段而 `ISAPI` 无法稳定覆盖

接入约束：

- 只允许落在后端模块
- 必须有明确业务收益和现场收益
- 必须形成单独文档、配置项和验收项
- 不得影响当前 `RTSP / MediaMTX / ONVIF / GStreamer` 主链稳定性

## 五、当前建议决策顺序

发现海康设备需求时，按下面顺序判断，不要直接跳到 `SDK`：

1. 先问是不是现有 `RTSP / ONVIF / GStreamer` 已能满足
2. 如果只是设备信息、编码信息、设备录像查询，优先考虑 `ISAPI`
3. 如果是事件订阅、设备端历史下载或私有能力，再评估 `SDK`
4. 只有在明确存在现场收益时，才允许把 `SDK` 纳入版本计划

## 六、当前版本结论

- `V1.1`：不引入海康 `SDK` 作为主播放链，继续维持现有视频主线
- `V1.1`：`ONVIF` 仍是 PTZ 与快照兜底主线
- `V1.1`：海康兼容性重点仍然是实机验收 `WebRTC / HLS / H.264 / H.265`
- `V1.3`：仅在设备端录像检索、设备端下载、事件订阅存在明确收益时，再单独评估 `ISAPI / SDK`
