## 2. 系统总体架构

### 2.1 物理架构

系统由车载感知单元、边缘计算服务器和远程监控终端三部分组成，所有设备通过工业以太网互联。边缘服务器采用 ARM Debian 11 原生服务部署，配置大容量 SSD 用于视频录像、点云快照和日志存储。

```

│  ┌───────────┐  │      │  ┌────────────────────┐  │      │  ┌───────────┐  │
│  │48线雷达   │──┼──────┼─▶│ARM Debian 11原生服务│  │      │  │PC/平板/   │  │
│  └───────────┘  │      │  │  - Qt后端服务       │  │      │  │手机浏览器 │  │
│  ┌───────────┐  │      │  │  - MySQL 服务       │◀─┼──────┼──┤           │  │
│  │400万球机  │──┼──────┼─▶│  - mediamtx 服务    │  │      │  └───────────┘  │
│  └───────────┘  │      │  │  - GStreamer管道    │  │      │                 │
│                 │      │  ├────────────────────┤  │      │                 │
│                 │      │  │ 2TB SSD (数据存储)  │  │      │                 │
└─────────────────┘      └───────────┬────────────┘  └─────────────────┘
                                   │ Modbus TCP/RTU
                              ┌─────▼─────┐
                              │ 行车PLC   │
                              │ (X, Y, θ) │
                              └───────────┘
```

### 2.2 逻辑架构

系统逻辑上分为五层：

| 层次 | 说明 |
|------|------|
| **感知层** | 雷达、球机、PLC（提供行车位姿） |
| **数据采集层** | 雷达驱动、视频采集、Modbus客户端 |
| **处理层** | 点云预处理、坐标转换、最高点检测、地图拼接、体积计算、录像管理 |
| **服务层** | WebSocket服务、HTTP服务、数据库服务、文件服务 |
| **表现层** | Web前端（Vue3 + Three.js） |

### 3.2 硬件安装与连接

#### 3.2.1 雷达安装
- **安装位置**：行车转盘下方或侧面，确保水平360°无遮挡。雷达中心与转盘旋转中心需精确测量偏移量。
- **安装角度**：建议向下倾斜5°~10°，使0°水平线对准料堆中部。
- **电气连接**：网线连接至交换机，供电12V DC（由服务器或独立电源提供）。

#### 3.2.2 球机安装
- **安装位置**：与雷达同轴或附近，视野覆盖抓斗作业区域。安装高度应高于抓斗最高点。
- **电气连接**：网线至交换机，支持POE供电或独立12V/24V。

#### 3.2.3 边缘服务器安装
- **安装位置**：行车电气室或现场控制柜，注意散热，避免阳光直射。
- **网络连接**：
  - 网口1：连接雷达、球机，配置独立子网（如192.168.111.x/24）。
  - 网口2：连接企业监控网络，可配置DHCP或静态IP。
- **串口扩展**：通过USB转RS485模块连接PLC（若PLC支持Modbus RTU）。
- **存储挂载**：2TB SSD 格式化为 ext4，挂载至 `/data` 或项目运行目录，用于视频录像、点云快照、日志和数据库备份持久化。

#### 3.2.4 与PLC连接
- 边缘服务器通过Modbus TCP（以太网）或Modbus RTU（RS485）与行车PLC通信，获取行车位姿并发送引导坐标。优先使用TCP，若PLC不支持则用RTU。

## 5. 核心算法设计

### 5.1 雷达极坐标转直角坐标

根据雷达说明书，转换公式如下：

```
X = r * cos(ω) * cos(-α) + R * cos(-α)
Y = r * cos(ω) * sin(-α) + R * sin(-α)
Z = r * sin(ω) + z
```

- r：实测距离（米）
- ω：垂直角（弧度，范围 -5°~+45°）
- α：水平角（弧度，范围 0~2π）
- R：接收光心到原点的平面半径（设备参数，单位米，可从雷达设备信息帧获取）
- z：光心到原点的Z轴高度（设备参数）

实现时，解析雷达UDP数据包，提取每个通道的r、ω、α，代入公式计算。

### 5.2 坐标系定义与变换

#### 5.2.1 坐标系定义
| 符号 | 名称 | 原点 | 轴向 |
|------|------|------|------|
| R系 | 雷达坐标系 | 雷达底座中心 | X前，Y左，Z上 |
| T系 | 转盘坐标系 | 转盘旋转中心 | 与R系平行 |
| G系 | 大车坐标系 | 大车固定点（如轨道中心与转盘中心垂线交点） | X沿轨道，Y垂直轨道，Z上 |
| W系 | 世界坐标系 | 料场固定原点（如轨道一端） | 同G系 |

#### 5.2.2 变换参数
- **雷达安装偏移** `t_r2t`：R系原点到T系原点的平移向量 (Δx, Δy, Δz)，需标定。
- **转盘角度 θ**：从PLC获取，定义T系相对于G系的旋转角（绕Z轴）。
- **大车位置** (X_g, Y_g)：从PLC获取，G系原点在W系中的平移。

#### 5.2.3 变换公式
1. **R → T**：`p_t = p_r + t_r2t`
2. **T → G**：`p_g = R_z(-θ) * p_t`，其中旋转矩阵为：
   ```
   [cosθ  sinθ  0]
   [-sinθ cosθ 0]
   [0     0    1]
   ```
   注意符号：转盘顺时针旋转为正时，θ为正，则点应逆时针补偿，故用 -θ。
3. **G → W**：`p_w = p_g + (X_g, Y_g, 0)`

总变换矩阵（R→W）：
```
T_r2w = T_g2w * R_z(-θ) * T_r2t
```

### 5.3 最高点检测算法

输入：G系点云（经过下采样）
输出：最高点坐标 (X_max, Y_max, Z_max) 在G系，同时计算W系坐标。

步骤：
1. **ROI裁剪**：以当前抓斗位置为中心，设定矩形区域（长宽可配置，如10m×10m），提取区域内点云。
2. **统计滤波**：使用 `pcl::StatisticalOutlierRemoval`，设置邻近点数为50，标准差倍数1.0，去除孤立噪点。
3. **体素下采样**：使用 `pcl::VoxelGrid`，设置体素边长0.05m，降低点数至数千。
4. **搜索最高点**：遍历点云，记录Z值最大的点。若点云为空，则返回无效。
5. **平滑**：保留最近5帧的最高点，取平均值作为最终输出，抑制抖动。
6. **计算绝对坐标**：将相对最高点通过当前位姿变换到W系（使用T_g2w）。

### 5.4 地图拼接算法

- 全局地图 `global_map` 为 `pcl::PointCloud<pcl::PointXYZ>` 类型，世界坐标系。
- 每帧W系点云 `frame_w` 加入 `global_map`，使用 `+=` 操作。
- 当 `global_map->size() > MAX_POINTS` (如200万) 时，执行体素滤波：
  ```cpp
  pcl::VoxelGrid<pcl::PointXYZ> voxel;
  voxel.setInputCloud(global_map);
  voxel.setLeafSize(0.05f, 0.05f, 0.05f);
  voxel.filter(*global_map);
  ```
- 每10帧保存一次全局地图至PCD文件，文件名格式：`map_YYYYMMDD_HHMMSS.pcd`。

### 5.5 体积计算算法

输入：全局地图、货场边界多边形
输出：各料堆体积列表

步骤：
1. **边界裁剪**：使用 `pcl::CropHull` 或自定义多边形裁剪，提取边界内点云。
2. **地面拟合**：
   - 方法A：高度直方图法。统计点云Z值，取最低5%点的平均高度作为地面高度 `h_ground`。
   - 方法B：RANSAC平面拟合。使用 `pcl::SACSegmentation` 拟合平面，取内点作为地面，平面高度作为 `h_ground`。
3. **料堆分割**：
   - 滤除地面点：`z > h_ground + 0.1` 的点为料堆候选。
   - 欧式聚类：使用 `pcl::EuclideanClusterExtraction`，设置聚类阈值0.5m，最小点数100，最大点数不限，分割出独立料堆。
4. **体素积分**：
   - 对每个聚类点云 `cluster`，建立体素哈希表 `std::unordered_set<uint64_t>`。
   - 体素边长 `voxel_size = 0.1` (可配置)。
   - 遍历 `cluster` 每个点，计算体素索引：
     ```cpp
     int ix = floor(p.x / voxel_size);
     int iy = floor(p.y / voxel_size);
     int iz = floor(p.z / voxel_size);
     uint64_t key = ((uint64_t)ix << 42) | ((uint64_t)iy << 21) | (uint64_t)iz;
     voxel_set.insert(key);
     ```
   - 体积 `volume = voxel_set.size() * voxel_size * voxel_size * voxel_size`。
5. **结果输出**：每个料堆记录编号（按顺序或用户定义）、体积、测量时间、置信度（基于点云密度评估，如体素填充率）。

## 6. PLC通信协议与数据交互

### 6.1 通信方式

- **首选**：Modbus TCP，边缘服务器作为客户端，PLC作为服务器，端口502。
- **备选**：Modbus RTU (RS485)，边缘服务器通过USB转485模块连接，波特率115200，8位数据，1位停止，无校验。

### 6.2 寄存器地址映射

寄存器均为保持寄存器（Holding Register），16位。浮点数使用IEEE754单精度格式，占用两个连续寄存器，大端序（高字节在前）。

#### 6.2.1 输入寄存器（PLC → 边缘服务器）

| 寄存器地址 | 数据类型 | 说明 | 单位 | 范围 |
|-----------|---------|------|------|------|
| 0-1       | float32 | 大车位置X（沿轨道方向） | 米 | 0~500 |
| 2-3       | float32 | 小车位置Y（垂直轨道方向） | 米 | 0~100 |
| 4-5       | float32 | 转盘旋转角度θ | 弧度 | -π~π 或 0~2π |
| 6         | uint16  | 状态字 | - | 按位定义 |
| 7-8       | uint32  | 时间戳（毫秒） | 毫秒 | 0~2^32-1 |

**状态字定义（bit）**：
- bit0：大车编码器有效
- bit1：小车编码器有效
- bit2：转盘编码器有效
- bit3：PLC运行状态（1=运行）
- 其余保留

#### 6.2.2 输出寄存器（边缘服务器 → PLC）

| 寄存器地址 | 数据类型 | 说明 | 单位 | 范围 |
|-----------|---------|------|------|------|
| 100-101   | float32 | 相对坐标最高点X | 米 | - |
| 102-103   | float32 | 相对坐标最高点Y | 米 | - |
| 104-105   | float32 | 相对坐标最高点Z | 米 | - |
| 106-107   | float32 | 绝对坐标最高点X | 米 | - |
| 108-109   | float32 | 绝对坐标最高点Y | 米 | - |
| 110-111   | float32 | 绝对坐标最高点Z | 米 | - |
| 112       | uint16  | 心跳计数器 | - | 0~65535 |
| 113       | uint16  | 系统状态字 | - | 按位定义 |

**系统状态字定义（bit）**：
- bit0：系统正常（所有模块OK）
- bit1：雷达连接正常
- bit2：球机连接正常
- bit3：点云处理正常
- bit4：最高点有效（最近检测到有效点）
- 其余保留

### 6.3 数据更新频率与同步机制

- **读取频率**：50ms周期（20Hz），边缘服务器定时读取输入寄存器，记录读取时的本地系统时间（CLOCK_REALTIME）。
- **写入频率**：100ms周期（10Hz），最高点检测完成后，写入输出寄存器。
- **时间对齐**：
  - 雷达帧自带PTP时间戳，系统需配置PTP同步（见6.4）。
  - 在点云处理线程中，获取雷达帧时间戳 `t_lidar`，从最近读取的PLC数据缓冲区中查找与 `t_lidar` 最接近的记录，要求时间差 <20ms，否则丢弃该帧。
  - 缓冲区大小：保留最近1秒的PLC数据（约20组）。

### 6.4 错误处理与冗余

- **通信超时**：连续3次读取失败（超时或异常），标记PLC离线，停止引导，Web界面报警。
- **数据校验**：对读取的浮点数进行范围检查，如X坐标超出轨道范围，则丢弃。
- **心跳**：心跳计数器每100ms递增，PLC可监控此值，若超过1秒未变化，认为服务器故障。
- **重连机制**：TCP连接断开后，自动尝试重连，间隔1秒。

### 6.5 PLC端对接要求

行车厂家需完成：
1. 在PLC内建立Modbus服务器，开放指定端口（TCP）或配置RTU从站地址。
2. 将大车、小车、转盘编码器实时值转换为浮点数，映射到输入寄存器。
3. 接受输出寄存器的写入，并将最高点坐标用于自动控制或驾驶室显示。
4. 提供寄存器地址确认表，配合现场联调。

## 7. 核心处理模块详细设计

### 7.1 雷达驱动与解析模块

- **类名**：`LidarDriver`
- **职责**：
  - 创建UDP socket，绑定端口5600（可配置）。
  - 接收数据包，解析协议，生成PCL点云。
  - 将点云放入环形缓冲区 `RingBuffer<PointCloudPtr>`。
- **关键实现**：
  - 解析函数 `parsePacket(const uint8_t* data, int len)` 根据雷达协议（见附录）提取每个点的极坐标。
  - 使用雷达内部参数R和z（从设备信息帧获取）进行极坐标转直角坐标。
  - 点云类型：`pcl::PointCloud<pcl::PointXYZI>`，强度存储反射率。
- **环形缓冲区**：容量10帧，线程安全（互斥锁或无锁队列）。

### 7.2 视频采集与编码模块

- **工具**：GStreamer管道，利用RK3588硬件编码器。
- **管道示例**：
  ```bash
  gst-launch-1.0 rtspsrc location=rtsp://admin:pass@192.168.1.100/stream ! \
    rtph264depay ! h264parse ! v4l2h264enc ! h264parse ! \
    mpegtsmux ! srtsink uri=srt://0.0.0.0:8888
  ```
- **说明**：
  - `rtspsrc` 从球机拉取RTSP流。
  - `v4l2h264enc` 使用RK3588硬件编码器（需安装rkmpp驱动）。
  - 输出 SRT 流至 mediamtx 原生服务，由 mediamtx 转换为 WebRTC/HLS 供前端播放。
- **进程管理**：作为主机进程运行（因需硬件直通），由systemd管理，异常重启。

### 7.3 坐标转换模块

- **类名**：`CoordinateTransformer`
- **输入**：雷达点云（R系）、当前位姿（X_g, Y_g, θ）
- **输出**：大车坐标系点云、世界坐标系点云
- **实现**：
  - 使用Eigen构造变换矩阵 `T_r2g` 和 `T_r2w`。
  - 调用 `pcl::transformPointCloud` 分别转换。

### 7.4 最高点检测模块

- **类名**：`HighestPointDetector`
- **输入**：大车坐标系点云、当前抓斗位置（可选）
- **输出**：最高点坐标（G系和W系）
- **方法**：见5.3节。
- **平滑**：`std::deque<Point>` 保存最近5帧最高点，每帧计算平均值。

### 7.5 地图拼接模块

- **类名**：`GlobalMapManager`
- **成员**：`pcl::PointCloud<pcl::PointXYZ>::Ptr global_map`
- **方法**：
  - `void addFrame(PointCloudPtr frame_w)`
  - `void saveMap()`
  - `PointCloudPtr getMap()`
- **线程安全**：使用 `std::mutex` 保护 `global_map`。

### 7.6 体积计算模块

- **类名**：`VolumeCalculator`
- **方法**：`std::vector<PileVolume> calculate(PointCloudPtr map, const Polygon& boundary)`
- **实现**：见5.5节。

### 7.7 Modbus通信模块

- **类名**：`ModbusClient`
- **基于**：Qt SerialBus（`QModbusTcpClient` / `QModbusRtuSerialClient`）
- **线程模型**：Qt 定时器驱动读写周期，底层请求使用 `QModbusReply` 异步接口并在模块内部同步等待结果。
- **读取线程**：
  ```cpp
  while (running) {
    sendReadRequest(inputRegisters, slaveId);
    record_time = get_current_time();
    push_to_buffer(record);  // 包含时间戳和数据
    usleep(50000); // 50ms
  }
  ```
- **写入**：外部调用 `writeOutputs(pose)`，写入寄存器100-113。

### 7.8 WebSocket服务模块

- **基于**：Qt WebChannel + Qt WebSockets
- **步骤**：
  1. 创建 `QWebSocketServer`，监听端口12345。
  2. 创建 `QWebChannel`，注册C++对象（如 `PointCloudProcessor`）。
  3. 当有WebSocket连接时，将 `QWebChannel` 与socket关联。
- **注册对象**：
  - 属性：`points` (QVariantList, 点云数组)，`highestPoint` (QVariantList)，`gantryPose` (QVariantList) 等。
  - 信号：`pointsUpdated`，`highestPointChanged`。

### 7.9 HTTP服务模块

- **基于**：libmicrohttpd 或 Qt HttpServer
- **API实现**：
  - 静态文件服务：根目录 `/var/www/html`。
  - REST API：解析URL和JSON，调用相应处理函数。
- **云台控制**：通过球机的ONVIF接口实现，封装为内部函数。

## 8. 硬盘录像机功能设计

### 8.1 功能概述

实现球机视频的自动连续录制、分段存储、索引管理和按时间回放。支持与点云历史地图同步回放。

### 8.2 录像策略

- **录制模式**：全天候24小时连续录制（可配置时间计划）。
- **分段存储**：每1小时生成一个MP4文件，文件名为 `HHMMSS.mp4`，存放于 `/data/videos/YYYYMMDD/`。
- **编码参数**：H.265，恒定码率2Mbps，分辨率1080p，帧率25fps。
- **存储容量**：2TB SSD，约可存储46天录像（按每天43GB计算）。超过保留天数自动删除最旧文件。

### 8.3 录像管理模块设计

#### 8.3.1 模块组成
- **GStreamer录制管道**：主机进程，负责从球机拉流、编码、分段写入文件。
- **索引维护**：Qt服务监听文件生成事件，记录到MySQL数据库。
- **清理线程**：定时检查磁盘空间和文件年龄，删除过期文件。

#### 8.3.2 GStreamer管道实现
使用 `splitmuxsink` 实现自动分段：
```bash
gst-launch-1.0 rtspsrc location=rtsp://... ! rtph264depay ! h264parse ! \
  splitmuxsink location=/data/videos/20260314/080000-%05d.mp4 max-size-time=3600000000000
```
- `max-size-time=3600000000000` 表示1小时（纳秒）。
- `location` 由应用先生成日期目录和时间前缀，`splitmuxsink` 只追加段号。
- 分段文件名建议保留 `HHMMSS-%05d.mp4` 结构，便于按时间清理和索引。

#### 8.3.3 索引数据库（MySQL）
表 `video_records` 结构：

| 列名 | 类型 | 说明 |
|------|------|------|
| id | INT AUTO_INCREMENT PRIMARY KEY | 自增ID |
| start_time | BIGINT | 开始时间戳（Unix毫秒） |
| end_time | BIGINT | 结束时间戳（Unix毫秒） |
| file_path | VARCHAR(255) | 相对路径（如 `20260314/080000.mp4`） |
| file_size | BIGINT | 文件大小（字节） |
| duration | INT | 时长（秒） |

**索引维护**：通过 `splitmuxsink` 的 `splitmuxsink-closed` 信号，当一段文件写完时，获取文件路径、起止时间，插入数据库。

#### 8.3.4 存储管理
- **磁盘监控**：每小时检查剩余空间，若低于10%则报警，并删除最早录像（按start_time升序删除，直至空间>20%）。
- **过期删除**：每日凌晨检查，删除超过设定保存天数（如30天）的录像文件和对应数据库记录。

### 8.4 录像查询与回放接口

**HTTP API**：

| 端点 | 方法 | 说明 | 参数 |
|------|------|------|------|
| `/api/videos/list` | GET | 查询录像列表 | start, end (ISO8601时间) |
| `/api/videos/play/{file_path}` | GET | 直接播放MP4文件 | 路径URL编码 |
| `/api/videos/hls/{file_path}/playlist.m3u8` | GET | 返回HLS播放列表 | 需预先切片 |

**查询示例**：
```
GET /api/videos/list?start=2026-03-14T08:00:00&end=2026-03-14T10:00:00
```
返回：
```json
{
  "videos": [
    {"file": "20260314/080000.mp4", "start": "2026-03-14T08:00:00", "end": "2026-03-14T09:00:00", "duration": 3600},
    {"file": "20260314/090000.mp4", "start": "2026-03-14T09:00:00", "end": "2026-03-14T10:00:00", "duration": 3600}
  ]
}
```

### 8.5 与点云同步回放设计

在历史回放页面，用户选择日期后，前端同时请求录像列表和点云地图列表。时间轴上显示两个轨道。当用户拖动进度条时：
- 视频跳转到对应时间（MP4支持字节范围请求，或使用HLS切片精确跳转）。
- 点云加载对应时间点的PCD文件（最接近时间戳的地图）。

若需要精确同步，可在点云文件名中嵌入时间戳（如 `map_20260314_083215.pcd`），前端根据时间匹配。

### 10.4 Qt数据库连接代码

```cpp
// DatabaseManager.h
#include <QSqlDatabase>
#include <QSqlQuery>
#include <QSqlError>
#include <QDebug>
#include <QSettings>

class DatabaseManager {
public:
    static DatabaseManager& instance() {
        static DatabaseManager instance;
        return instance;
    }

    bool initialize() {
        // 从配置文件读取连接参数
        QSettings settings("/app/config/db.ini", QSettings::IniFormat);
        QString host = settings.value("db/host", "127.0.0.1").toString();
        int port = settings.value("db/port", 3306).toInt();
        QString dbName = settings.value("db/name", "grab_system").toString();
        QString user = settings.value("db/user", "grab_user").toString();
        QString password = settings.value("db/password").toString();

        db = QSqlDatabase::addDatabase("QMYSQL");
        db.setHostName(host);
        db.setPort(port);
        db.setDatabaseName(dbName);
        db.setUserName(user);
        db.setPassword(password);

        if (!db.open()) {
            qCritical() << "Database connection failed:" << db.lastError().text();
            return false;
        }

        qInfo() << "Database connected successfully";
        return true;
    }

    QSqlDatabase getConnection() { return db; }

private:
    DatabaseManager() = default;
    QSqlDatabase db;
};
```

### 13.1 雷达UDP数据包解析

参考雷达说明书V1.0，点云数据帧格式如下：

- **UDP包头**（42字节）：
  - 帧头：4字节，固定为 `0xBB 0xCC 0x55 0x66`
  - 帧计数：2字节
  - 包计数：2字节
  - 时间戳-秒：6字节
  - 时间戳-亚秒：4字节
  - ...（详见说明书）

- **数据块**：每个通道10字节，结构：
  - 水平角：2字节，单位0.01°
  - 距离：2字节，单位0.009368m
  - 反射率：1字节
  - 脉宽：1.5字节（12位）
  - 置信度：0.5字节（4位）
  - ...（详见说明书）

解析时需注意字节序（大端），以及通道数（48线，双回波时96通道）。

### 13.2 坐标转换标定流程

1. 将行车停靠在已知参考点（如轨道零点），记录PLC读数（X_g, Y_g, θ=0）。
2. 在雷达前方约10米处放置角反射器，用全站仪测量其在世界坐标系下的坐标 `P_w`。
3. 从雷达点云中找到角反射器的点，获取其在雷达坐标系下的坐标 `P_r`。
4. 根据变换公式 `P_w = T_g2w * R_z(0) * T_r2t * P_r`，已知 `P_w`, `P_r`, `T_g2w`（由X_g,Y_g确定），可求解 `T_r2t`。
   - 先忽略旋转，解平移：`P_w - (X_g, Y_g, 0) = P_r + t_r2t`，得 `t_r2t` 近似值。
   - 精确求解可用最小二乘法多点标定。
5. 移动行车至另一位置，重复测量验证，若偏差大则重新标定。

### 13.3 Modbus通信测试工具

- **Modbus Poll**：Windows下模拟客户端，用于测试PLC寄存器。
- **QModMaster**：开源跨平台工具。
- **Python脚本示例**：
  ```python
  from pymodbus.client import ModbusTcpClient
  client = ModbusTcpClient('192.168.1.10', port=502)
  client.connect()
  # 读取输入寄存器0-5 (3个float)
  rr = client.read_holding_registers(0, 6)
  if not rr.isError():
      import struct
      x = struct.unpack('>f', struct.pack('>HH', rr.registers[0], rr.registers[1]))[0]
      y = struct.unpack('>f', struct.pack('>HH', rr.registers[2], rr.registers[3]))[0]
      theta = struct.unpack('>f', struct.pack('>HH', rr.registers[4], rr.registers[5]))[0]
      print(f"X={x}, Y={y}, θ={theta}")
  client.close()
  ```
