# 视频现场回写顺序卡

这份顺序卡只做一件事：让现场按固定顺序完成 `V1.1` 视频链路回写，不再临时发挥。

适用范围：

- 海康摄像机
- 当前项目视频主线
- 真实现场联调

## 1. 先看状态，不要先点按钮

进入监控页后固定按这个顺序看：

1. `实时链路`
2. `链路异常`
3. `MediaMTX`
4. `端口探测`
5. `配置源`
6. `视频基线`
7. `浏览器兼容`
8. `视频自检`
9. `录像前提`
10. `录像健康`
11. `插件 / 缺失插件`
12. `录像编码`

如果前 6 项已经异常，不继续做截图、录像、回放结论。

## 2. 再做动作验证

固定顺序：

1. 实时预览
2. 本地截图
3. 后端抓图兜底
4. 开始录像
5. 停止录像
6. 回放录像
7. PTZ `stop`
8. PTZ `presets`
9. PTZ `goto preset`
10. PTZ `move`
11. PTZ `relative-move`
12. PTZ `home`

## 3. 每一步至少记这几个结论

### 实时预览

- 默认是否稳定命中 `WebRTC`
- 是否回退到 `HLS`
- 当前浏览器
- 当前编码

### 抓图

- 本地截图是否成功
- 后端 `/api/video/snapshot` 是否成功
- 失败时的真实错误

### 录像

- `recording/status` 是否就绪
- `healthStatus / recoveryStatus`
- 文件是否可播放
- 文件是否可回放

### PTZ

- `wsSecurityEnabled`
- 方向是否正确
- 预置位是否稳定

## 4. 联调后必须回写到哪里

每次联调结束后，至少同步这三处：

1. [`DOC/当前现场验收包/视频实机验收矩阵.md`](当前现场验收包/视频实机验收矩阵.md)
2. [`DOC/当前现场验收包/现场联调验收记录.md`](当前现场验收包/现场联调验收记录.md)
3. `rtk bash scripts/arm/verify_remaining_acceptance_closure.sh`

如果结论变更了浏览器兼容建议，再补：

4. [`视频浏览器兼容矩阵`](视频浏览器兼容矩阵.md)

如果结论涉及海康协议边界或设备差异，再补：

5. [`海康SDK与ISAPI能力边界`](海康SDK与ISAPI能力边界.md)
6. [`海康设备联调SOP`](海康设备联调SOP.md)
7. 如果第 3 步返回 `PASS`，执行 `rtk bash scripts/arm/finalize_remaining_acceptance_closure.sh`

## 5. 当前禁止直接下结论的情况

遇到下面这些情况，只能记为“待确认”，不能直接写“通过”：

- 没有真实摄像头环境
- 没有回写 `H.264 / H.265` 实机矩阵
- 只测了 `HLS`，没测 `WebRTC`
- 只测了截图，没测录像和回放
- 只测了 PTZ 移动，没测预置位和停止

## 6. 当前版本结论

- 这张顺序卡是 `V1.1` 视频链路验收执行入口
- 第 `11 / 12 / 13 / 20` 项仍需真实现场回写后才能关闭
