# Claude Rate Limit Auto Continue Contract

这是给后续 Codex、Claude 或人工维护者使用的实现契约。

目标不是规定每个平台具体怎么按键，而是规定这套自动继续流程必须达成什么结果、按什么顺序执行，以及如何判断成功。

## 目标

- 识别 Claude 的 `StopFailure`
- 从失败信息中提取限额重置时间
- 在到点前保留缓冲
- 到点后自动输入 `继续` 并回车
- 全程可记录日志、可去重、可复用

## 支持平台

- 原生 macOS
- 原生 Linux
- 原生 Windows
- WSL

## 统一要求

- 同一套流程在不同平台上保持一致
- 业务逻辑与平台输入模拟分离
- 解析逻辑、等待逻辑、去重逻辑必须独立于平台实现
- 平台相关能力只允许出现在输入模拟层或通知层
- 不允许把平台差异混进限额解析逻辑里

## 输入

Claude hook 会提供一段 JSON 输入。实现必须从中尽量提取以下字段：

- `last_assistant_message`
- `error`
- `error_details`
- `reason`
- `session_id`

如果字段结构变化，优先兼容字段内容，而不是依赖固定层级。
如果 JSON 解析失败或字段缺失，必须把原始 hook 输入当作兜底文本继续解析。

## 输出

实现必须能产出以下结果：

- 一条可读日志，说明检测到限额以及预计等待时长
- 一个去重机制，避免同一会话重复启动多个等待任务
- 到点后的自动输入动作
- 一条成功日志，说明已经输入 `继续` 并回车

## 流程

1. Claude 触发 `StopFailure`
2. Hook 入口读取事件输入
3. 从输入文本中提取重置时间
4. 如果没有提取到重置时间，则只写日志并退出
5. 如果已经有同一会话、同一重置时间的任务，则直接退出
6. 创建锁并启动后台等待任务
7. 后台任务等待到重置时间附近
8. 预留固定缓冲时间
9. 自动输入 `继续`
10. 执行回车
11. 写入完成日志并清理锁

## 时间规则

- 如果能解析出明确的绝对时间，就以这个时间为准
- 支持常见的本地时间和 ISO 8601 时间格式
- 如果重置时间已经过去很久，应该放弃而不是无限等待
- 预留缓冲时间是流程的一部分，不能省略

## 平台适配原则

平台适配只负责“怎么把 `继续` 输入到当前前台应用里”，不负责判断什么时候输入。

允许的做法包括但不限于：

- 剪贴板 + 粘贴 + 回车
- 键盘输入模拟
- 平台原生自动化工具

实现者可以按平台选择最稳定的方式，但必须满足以下结果：

- 当前前台 Claude 输入框最终收到 `继续`
- 随后触发一次回车
- 过程对用户是自动的
- 粘贴和回车之间需要保留一个短暂间隔，避免平台抢占输入焦点

## 日志要求

- 必须有开始日志
- 必须有未解析到时间的日志
- 必须有重复任务的日志
- 必须有完成日志
- 日志文件位置应固定且易查

## 验收标准

这套流程只有在以下条件都满足时才算完成：

- 能在真实 `StopFailure` 上触发
- 能正确提取限额重置时间
- 能跨平台执行到最后一步
- 能在到点后自动输入 `继续` 并回车
- 能通过日志确认整个过程

## 推荐的实现边界

- `rate_limit_continue.sh`
  - 只负责 hook 输入、解析、去重、启动后台任务
- `rate_limit_continue_worker.py`
  - 只负责等待、通知和最终自动输入
- 平台专属实现
  - 只负责输入和通知，不改变流程判断

## 维护方式

以后如果要改这套流程，优先按以下顺序处理：

1. 先改契约，再改实现
2. 先保证行为一致，再优化平台细节
3. 任何平台新方案都必须满足当前验收标准

## 给 Agent 的说明

如果你是 Codex 或 Claude，且你要改这套工具，请遵循这份契约：

- 不要改业务流程定义
- 不要把平台细节扩散到解析层
- 不要删除日志和去重
- 不要在未更新契约的情况下改验收标准
- 完成后同步更新 README 和索引
