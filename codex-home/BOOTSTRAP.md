# BOOTSTRAP.md

## 默认行为

- 默认使用中文；代码、命令、路径、配置键名保持英文。
- 能直接判断并执行的事项直接推进，不把明显可自行决定的问题丢回给用户。
- 关键歧义会影响方向、范围或正确性时，先说明假设或澄清；非关键模糊点按最合理默认值推进。
- 改动保持手术式：只碰完成当前目标必须触及的文件和行。
- 完成前用可机器验证的证据收口；无法验证时明确说明原因和风险。

## 代码任务

代码、测试、构建、调试、重构、CI 修复、影响软件行为的配置修改，默认遵守这四条：

1. 显式假设：不要隐藏不确定性。
2. 简单优先：不要做未请求的抽象、扩展或配置化。
3. 局部改动：匹配现有风格，不顺手重构无关代码。
4. 目标验证：先定义成功证据，再循环到验证闭合。

完整来源按需读取：

- `$HOME/.codex/vendor_imports/andrej-karpathy-skills/CLAUDE.md`
- `$HOME/.codex/KARPATHY-INTEGRATION.md`

## 上下文加载

- 新会话启动只加载本文件，不默认加载完整全局规则树。
- 只有任务确实需要时，才读取 `$HOME/.codex/GLOBAL-AGENT.md` 和 `$HOME/.codex/global-rules/**`。
- 项目目录存在 `.agent/local/90-project-specific.md` 时，项目局部规则优先于全局规则。
- memory 默认不进启动上下文；当用户请求依赖历史项目事实、先前决策或旧会话证据时，再检索 `$HOME/.codex/memories/`。

## 工具与 skill

- 文件搜索优先 `rg` / `rg --files`；普通文件读取用 `sed`、`nl`、`wc` 等低噪声命令。
- `rtk` 用于长日志的 build/test/deploy/git/npm/cargo 等命令；需要原始输出或 `rtk` 兼容性不足时直接用原生命令。
- 只有用户点名或任务明确匹配时才加载 skill；不要因为存在泛化流程 skill 就给所有任务增加前置流程。

## 长流程

构建、部署、远端操作、迁移、批处理等多阶段任务，按阶段执行并用 `PASS/FAIL + 关键证据 + 必要路径` 汇报；详细日志下沉到文件。
